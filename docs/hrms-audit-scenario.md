# SCENARIO — New hire, end to end

**Scope.** One candidate, one offer, one hire: offer acceptance at `/portal/offer/[token]` → `hr_employees` creation → joining-document submission at `/portal/onboarding` → HR review at `/admin/hr/onboarding` → Face-2FA enrolment and the profile photo it writes.

**Evidence basis.** Steps below are drawn from two audited domains that survived an adversarial verification pass (46 and 35 confirmed claims respectively), with the verifier's corrections applied. The Face-2FA/profile-photo domain arrived truncated, so steps 17-20 were re-read directly from source (`src/middleware.ts`, `src/pages/portal/face-setup.astro`, `src/pages/api/auth/enroll-face-selfie.ts`, `src/pages/api/auth/enroll-face.ts`). Anything the verifier could not confirm is marked **[UNVERIFIED]** and must not be relied on.

**Two structural warnings before the walkthrough:**

1. **No transaction anywhere in this scenario.** The acceptance chain (steps 6-9) is four independent statements. Every downstream write is likewise standalone. There is no `db.transaction(...)` on any path described here.
2. **The sequence below is the intended order, not the observed one.** The Face-2FA gate (step 2) fires on the *first* protected page the account ever loads — in practice long before offer acceptance. Because of that, the profile-photo write in step 20 is normally unreachable for a real hire (see step 20 and Gap G1).

## Walkthrough

### Phase 0 — Preconditions

---

**1. HR issues the offer (upstream context)**

- **Actor:** HR/admin user on `/admin/applications/[id]/offer-letter`
- **Trigger:** POST from the offer-letter builder
- **System behaviour:** Creates/updates the `offer_letters` row with `status = 'sent'`, advances `applications.status` to `'offer'`, and provisions a login account via `createUserAccount`, whose id is stored as `offer_letters.created_user_id`. For a **brand-new** account the role is hard-coded `let role: any = 'editor'` (`offer-letter.astro:69`); for a candidate who already applied through the portal the existing account is reused and its role stays `'applicant'` (`offer-letter.astro:54-65`).
- **Writes:** `offer_letters` (token, status, content jsonb, candidate_name, candidate_email, role_title, joining_date, expiry_date, created_user_id, …); `applications.status`; `users` (insert or reuse).
- **Validation:** Not re-audited at this depth.
- **Authorization:** Admin surface behind the middleware admin gates.
- **Failure mode:** A candidate is granted the staff role `editor` **at offer-send time**, before they have accepted or declined anything.
- **[UNVERIFIED]** The offer domain's endpoint listing was truncated mid-entry in the audit submission; only `GET /portal/offer/[token]` was independently evaluated. Treat this step's authz description as unconfirmed.

---

**2. Face-2FA gate fires on the first protected page**

- **Actor:** Middleware (`src/middleware.ts`)
- **Trigger:** Any request where `isProtected(path)` is true (`/admin`, `/admin/*`, `/portal`, `/portal/*`, `/hei/portal*` — lines 96-104) and `isExempt(path)` is false (lines 72-92; `/api/*`, `/_astro/*`, all login and enrolment surfaces are exempt).
- **System behaviour:** `hasFaceEnrolled(user.id)` runs `SELECT id FROM user_face_enrollments WHERE user_id = $1 AND is_active = true LIMIT 1`. On no row: `role === 'applicant'` → 302 to `/portal/face-setup?next=<path>`; **every other role** → 302 to `/enroll-face` (lines 279-292).
- **Writes:** Session cookie refresh (`httpOnly`, `sameSite 'lax'`, `secure` in prod); possible one-time `UPDATE users SET access_status='approved'` for pending applicants, guarded by an `era_appr` cookie.
- **Validation:** Session validity plus existence of an active enrolment row.
- **Authorization:** This *is* the gate.
- **Failure mode:** The destination differs by role and the two destinations do materially different things (steps 17-20). No page in this scenario is reachable until enrolment completes.

### Phase 1 — Acceptance

---

**3. Candidate opens the offer**

- **Actor:** Signed-in user
- **Trigger:** `GET /portal/offer/<token>`
- **System behaviour:** `if (!user) redirect('/portal/login')`. Loads the offer by token; missing → `/portal`. Ownership check: `applications.id = offer.applicationId AND applications.applicant_user_id = user.id`. Then `const isAdmin = user.role !== 'applicant'` (`[token].astro:32`) — **any** non-applicant role skips the ownership check entirely.
- **Writes:** None. `markOfferOpened` is imported at line 4 and never called, so `opened_at` / `open_count` / `last_opened_ip` are never written from this page.
- **Validation:** Session; token row exists; ownership (bypassed when `role !== 'applicant'`).
- **Authorization:** Session + ownership, or role-negation bypass.
- **Failure mode:** Any holder of a token with any non-applicant role — including `marketing` and the AquinTutor scopes — reads the full offer, the signature image, the signature IP and the user agent. This is a role negation, not a permission check.

---

**4. Candidate signs**

- **Actor:** Candidate
- **Trigger:** Types their legal name, ticks the consent box, submits the form (POST to the same URL)
- **System behaviour:** Client JS (`nameToSignatureImage`) rasterises the typed name to a cursive PNG data URL into the hidden `signatureDataUrl` input (SVG data-URL fallback if canvas fails). The handler reads the body **twice**: `await Astro.request.clone().formData()` at line 38 to peek at `action` (39) and, on decline, `declineReason` (49); then `await Astro.request.formData()` at line 93. Inside the sign branch only `signatureDataUrl` is read.
- **Writes:** None yet.
- **Validation:** `signatureDataUrl` non-empty, `startsWith('data:image/')`, length ≤ 500000. That is the entire validation.
- **Authorization:** `if (Astro.request.method === 'POST' && !isAdmin)` (line 37) — admins are read-only here. No CSRF token; `astro.config.mjs` sets `security: { checkOrigin: false }` globally and the session cookie is `sameSite 'lax'`.
- **Failure mode:** `typedName` is submitted but never read server-side. The consent checkbox `#acceptCheck` (line 511) has **no `name` attribute**, so it is never in the request body at all; the only enforcement is client-side JS at 635-639. The persisted "signature" is a client-generated image the client fully controls.

**5. Status gate on signing**

- **Actor:** SSR POST handler
- **Trigger:** Sign branch entry
- **System behaviour:** Rejects only `offer.status === 'signed'` and `'withdrawn'` (lines 88-91). `'draft'`, `'declined'` and any expired offer fall through. `expiryStatus`/`daysLeft` are computed at 194-202 and used **only** to decide whether the form renders (line 490).
- **Writes:** None.
- **Validation:** Two statuses.
- **Authorization:** As step 4.
- **Failure mode:** An expired offer, a draft offer, or one the candidate already declined can be signed — flipping the application from `rejected` back to `hired` and creating an employee. **Reachability qualifier (verifier correction):** the accept form renders only when `status === 'sent'` and not expired and not withdrawn (line 490), and the submit handler binds only `if (sigForm && isAvailable && !isAdmin)` (line 626). Exploiting this requires a hand-crafted POST, not the rendered UI. The decline branch, by contrast, correctly checks all of signed/withdrawn/declined (42-47).

**6. Signature recorded**

- **Actor:** System
- **Trigger:** Validation passed
- **System behaviour:** Reads `x-forwarded-for` (falling back to `cf-connecting-ip`) and `user-agent`.
- **Writes:** `UPDATE offer_letters SET status='signed', signed_at=now(), signature_data_url=<client image>, signature_ip=<ipHeader.substring(0,64)>, signature_user_agent=<ua>, updated_at=now() WHERE id = offer.id`
- **Validation:** None beyond step 4.
- **Authorization:** As step 4.
- **Failure mode:** `offer_letters.status` is a plain `varchar(20)` with no enum or CHECK — `'signed'` has no DB-level meaning. `signature_ip` truncates a spoofable, possibly comma-separated XFF chain at 64 chars, potentially mid-address. The UI nevertheless tells the candidate "My signature, IP address, and timestamp will be permanently recorded as evidence of acceptance."

**7. Application advanced to hired**

**9. Applicant converted to employee**

- **Actor:** System → `transferApplicantToEmployee` (`src/lib/hire-transfer.ts`)
- **Trigger:** Dynamic import inside `try { ... } catch (_) {}` at `[token].astro:132-136`
- **System behaviour:** De-duplicates first (lines 18-24): `SELECT id FROM hr_employees WHERE (<appId>::uuid IS NOT NULL AND application_id = <appId>) OR (<userId>::uuid IS NOT NULL AND user_id = <userId>) OR lower(personal_email) = <email> OR lower(email) = <email> LIMIT 1`; an existing row returns `{created:false, employeeId}` and writes nothing. Employee code: `'EMP-' + new Date().getFullYear() + '-' + Math.random().toString(36).slice(2,7).toUpperCase()`. **Verifier correction:** values are not content-only — name, email and designation each fall back to the offer row (`c.candidateName || offer.candidateName || offer.candidate_name || 'New Employee'` line 13; email line 14; `roleTitle` line 27), and `joining` is coerced to SQL NULL when blank (line 29), so the insert is `${joining}::date` with a possible NULL. Only `employment_type` (`c.employmentType || ''`) and `phone` (`c.candidatePhone || null`) are content-only.
- **Writes (lines 33-36):** `INSERT INTO hr_employees (user_id, employee_code, full_name, email, personal_email, phone, designation, employment_type, joining_date, employment_status='active', application_id, is_active=true, onboarding_status='pending') RETURNING id`
- **Validation:** `if (!email) return { created: false }` (line 15).
- **Authorization:** Inherited.
- **Failure mode:** Triple-swallowed. The inner catch logs `[hire-transfer] failed:` and returns `{created:false}`; the outer `catch (_) {}` discards even that; the return value is never inspected. A blank candidate email, an employee-code collision, or any DB error produces **no employee row** while the application is already `'hired'` and the candidate is told they are hired. `employee_code` has ~60M combinations, no uniqueness retry.

### Phase 2 — Joining documents

---

**12. Hire loads the document page**

- **Actor:** Any signed-in user
- **Trigger:** `GET /portal/onboarding` (`prerender = false`)
- **System behaviour:** `if (!Astro.locals.user) redirect('/portal/login?next=/portal/onboarding')`. Then `docs = await listDocs(String(user.id)).catch(() => [])` (`SELECT * ... WHERE user_id = $uid ORDER BY id ASC`), `p = progress(docs)`, `remaining = Math.max(0, 5 - docs.length)`. Chips render as Verified / "Needs another copy" (status `rejected`) / "Under review". The add form renders only when `remaining > 0`; otherwise: **"You have submitted all 5 documents. Remove one if you need to replace it."** (`onboarding.astro:78` — verifier correction; the audit had truncated the second sentence).
- **Writes:** None, except the lazy DDL inside `ctx()` (`CREATE TABLE IF NOT EXISTS` + two `CREATE INDEX IF NOT EXISTS`) which runs once per process.
- **Validation:** Sign-in only.
- **Authorization:** Session + Face-2FA gate. **No check that an offer was accepted, no employment record check, no role restriction.**
- **Failure mode:** `.catch(() => [])` swallows read errors — a DB outage renders as "Nothing submitted yet." and re-opens the form, telling a hire they have zero documents when they may have five.

**13. Hire submits a Drive link**

- **Actor:** New hire
- **Trigger:** Selects `docType` (6 options), optional `title` (maxlength 200), `driveUrl` (`input type=url` required), clicks Submit
- **System behaviour:** Inline script (block at `onboarding.astro:85-113`, handler at 92-103; lines 64-75 are the form markup) preventDefaults, disables the button, POSTs JSON `{action:'add', docType, title, driveUrl}` to `/api/portal/onboarding/documents` with `credentials:'same-origin'`, then `location.reload()` after 700ms.
- **Writes:** Nothing client-side.
- **Validation:** Browser-side only — `required`, `type=url`, `maxlength=200`.
- **Authorization:** n/a client-side.
- **Failure mode:** The entire submit path is client JS; with JS disabled nothing useful happens.

**14. Server validates and inserts**

- **Actor:** `POST /api/portal/onboarding/documents` → `addDoc` (`src/lib/hr-onboarding.ts:87-99`)
- **Trigger:** `action: 'add'`
- **System behaviour, in exact order:** (1) `linkProblem(driveUrl)`; (2) `isDriveLink(driveUrl)`; (3) `if (await countDocs(userId) >= 5)` — the only server-side cap; (4) unknown `docType` silently coerced to `'other'`; (5) INSERT with `title.slice(0,200)` and `driveUrl.trim()`. `employeeId` is never passed by the API, so `employee_id` is always NULL.
- **Writes:** `INSERT INTO hr_onboarding_documents (user_id, employee_id=NULL, doc_type, title, drive_url)`; defaults fill `status='submitted'`, `created_at=now()`; `review_note` / `reviewed_by` / `reviewed_at` stay NULL.
- **Validation:** `isDriveLink` = `/^https:\/\/(drive|docs)\.google\.com\/[^\s]+$/i` and length ≤ 500. Rejection messages (verifier correction — the source uses **em dashes**, not hyphens): empty → "Paste the Google Drive link to the document."; non-https → "The link must start with https:// — copy it straight from Drive." (`hr-onboarding.ts:31`); >500 chars → "That link is unusually long — paste the plain share link." (`:33`); non-Drive host → "Please share the document from Google Drive (drive.google.com or docs.google.com)." (`ACCESS_FORMAT` at line 20 genuinely uses a plain hyphen: "Anyone with the link - Viewer".)
- **Authorization:** Authenticated session only. No role check on `add`.
- **Failure mode:** `countDocs`-then-INSERT is non-atomic with no backing constraint, so concurrent requests store 6+ rows. No de-duplication — the same URL can be submitted five times. The server never fetches the URL, so a private, deleted or wrong-file link is accepted and only discovered when HR clicks through. `addDoc` failures return **HTTP 200** with `{ok:false, error}`; unhandled exceptions go to `trackError('onboarding.documents_failed', ...)` and also return HTTP 200.

**15. Hire removes a document**

- **Actor:** New hire
- **Trigger:** "Remove" (rendered only when `d.status !== 'verified'`), then `confirm()`
- **Writes:** `DELETE FROM hr_onboarding_documents WHERE id = $id AND user_id = $uid AND status <> 'verified' RETURNING id` — hard delete, no soft-delete, no archive.
- **Validation:** Ownership and status enforced **inside the SQL predicate** — a genuine IDOR guard.
- **Authorization:** Session; scoping is in the WHERE clause.
- **Failure mode:** Rejected documents are deletable, so a hire can erase the evidence of an HR rejection and its `review_note` permanently. A non-existent id, another user's id and a verified row all return the same message: "That document is already verified and can no longer be removed."

### Phase 3 — HR verification

---

**16. HR opens the review queue**

- **Actor:** HR / staff
- **Trigger:** `GET /admin/hr/onboarding`
- **System behaviour:** Middleware: applicants → `/portal` (line 238); non-applicants without `can(user,'admin.access')` → `/aquintutor/admin/*` (line 252); `resolveAdminSection('/admin/hr/onboarding')` resolves via the longest-prefix entry `['/admin/hr','hr']`, so a custom role lacking `'hr'` in `getViewableSectionKeys` → `/admin?denied=hr` (lines 63-68, 265-273); then the Face-2FA gate. Page-level: no user → `/admin/login`; role `applicant` → `/portal`. Then `docs = await pendingForReview(150).catch(() => [])`, `waiting = docs.filter(d => d.status==='submitted').length`.
- **Writes:** None.
- **Validation:** Session + non-applicant + `admin.access` + `hr` section + face 2FA.
- **Authorization:** As above — this is the strongest gate in the scenario, and step 17 does not use it.
- **Failure mode:** Fixed `LIMIT 150` ordered by `(status='submitted') DESC, id DESC`, with no pagination, filter, search or grouping by hire, and no indication of truncation. The "Documents total" counter is the length of that capped page, not a true total. `pendingForReview` joins `LEFT JOIN users u ON u.id::text = d.user_id` — the cast defeats the users PK index.

**17. HR verifies or sends back**

- **Actor:** HR reviewer (nominally)
- **Trigger:** "Verify", or "Send back" after a native `window.prompt('Why is this being sent back? The hire will see this.')`
- **System behaviour:** POST `{action:'review', id, status, note}` to the **same public endpoint** `/api/portal/onboarding/documents`. Status normalisation: `'verified'` → `'verified'`, `'rejected'` → `'rejected'`, **anything else → `'submitted'`**. Note is `String(b.note).slice(0,300)`, passed as `undefined` when falsy, which stores NULL and clears any previous note. Response is `{ok:true}` only; the page reloads.
- **Writes:** `UPDATE hr_onboarding_documents SET status=$status, review_note=$note|NULL, reviewed_by=$uid, reviewed_at=now() WHERE id = $id`
- **Validation:** None on the status value beyond the coercion above.
- **Authorization:** Exactly `if (!user.role || user.role === 'applicant') return 403` (`documents.ts:29`). Since `users.role` is NOT NULL with default `'applicant'`, the `!user.role` half is dead code.
- **Failure mode — the most serious finding in this scenario.** The `WHERE` clause is `WHERE id = $id` with no scoping, and ids are sequential bigserial. Any signed-in non-applicant — including `partner`, `teacher`, `technical_moderator`, roles middleware explicitly bans from `/admin` — can verify or reject any hire's credential by guessing an id. **Verifier correction on the mechanism:** the audit blamed `isExempt()` waving through `/api/*`. That is wrong. `isExempt()` is consulted only at `middleware.ts:279`, for the face gate. The three admin guards (238, 252, 265) are keyed on `path === '/admin' || path.startsWith('/admin/')`, so this endpoint was never in their scope regardless of `isExempt`. The conclusion stands; the stated mechanism did not.

**18. Hire sees the result**

- **Actor:** New hire
- **Trigger:** Returns to `/portal/onboarding`
- **System behaviour:** Updated chip plus, when `review_note` is set, a red "HR note: …" line. `progress(docs)` = `{submitted: docs.length, verified, rejected, complete: docs.length > 0 && rejected === 0 && verified === docs.length}` (`hr-onboarding.ts:121-125`). When complete: "All your documents are verified. HR will be in touch about your joining formalities."
- **Writes:** None.
- **Failure mode:** **No notification of any kind on rejection** — no email, no in-app alert. The hire learns of a send-back only by reloading. `complete` is a pure function of the rows present: one verified document of type `'other'` marks a hire complete; no document type is mandatory and `DOC_TYPES` carries no required flag.

**19. Next portal visit routes the hire somewhere else**

- **Actor:** New hire
- **Trigger:** `GET /portal`
- **System behaviour:** `/portal/index.astro` bounces non-applicants to `/admin`, then runs `SELECT id, onboarding_status FROM hr_employees WHERE (personal_email = <email> OR email = <email> OR user_id = <id>) AND is_active = true LIMIT 1`. Because step 9 set `onboarding_status = 'pending'`, the user is redirected to `/portal/employee/onboarding` — a **different page** from the `/portal/onboarding` the acceptance CTA pointed at. That page collects bank details, govt ID, PAN and emergency contact and UPDATEs `hr_employees`.
- **Writes:** No write on redirect; `/portal/employee/onboarding` updates `hr_employees` on submit.
- **Validation:** `is_active = true` plus a matching email or user_id.
- **Authorization:** Session; must resolve to an `hr_employees` row, else `/portal`.
- **Failure mode:** The whole lookup sits in `try { ... } catch(_) {}`. Neither onboarding page links to the other, and nothing sets `onboarding_status` to complete when joining documents are submitted or verified.

**20. Applicant-role hire enrols and the photo is captured**

- **Actor:** User whose `users.role === 'applicant'`
- **Trigger:** 302 from step 2 to `/portal/face-setup?next=<path>`
- **System behaviour:** `if (!user) redirect('/portal/login')`. If an active enrolment exists and `?force=1` is absent, **redirect straight to `next`** (lines 12-20). Otherwise the page loads `@vladmandic/face-api@1.7.13` **from `https://cdn.jsdelivr.net`** (line 26), opens the camera, and collects up to 3 descriptors in ≤10 attempts, discarding any sample >0.65 Euclidean from the first, then averages them; it proceeds with as few as **one** sample (191-211). A code comment at 182-184 records that the blink liveness gate was **removed** as unreliable. The enrolment frame is then re-drawn un-mirrored into a 320×320 canvas and encoded as JPEG q0.82 (215-224) and POSTed with the descriptor to `/api/auth/enroll-face-selfie`.
- **Writes (API, `enroll-face-selfie.ts`):**
  - `INSERT INTO user_face_enrollments (user_id, face_descriptor, device_info, is_active) VALUES (...) ON CONFLICT (user_id) DO UPDATE SET face_descriptor=EXCLUDED.face_descriptor, device_info=EXCLUDED.device_info, is_active=true, enrolled_at=NOW()`
  - Then, **only if** `SELECT id FROM hr_employees WHERE user_id = $uid AND is_active = true` returns a row: `ALTER TABLE hr_employees ADD COLUMN IF NOT EXISTS photo_url TEXT` (executed **on the request path**), `UPDATE users SET photo_url = $url`, `UPDATE hr_employees SET photo_url = $url WHERE user_id = $uid`. If `BLOB_READ_WRITE_TOKEN` is set the image is uploaded to `employee-photos/<userId>-<ts>.jpg` with `access: 'public'` and the blob URL stored; otherwise **the base64 data URL itself is written into the column**.
- **Validation:** 401 without a session; 400 on invalid JSON; `Array.isArray(descriptor) && descriptor.length === 128`; rejects all-zero descriptors (`Math.abs(n) > 1e-6` on at least one element). Selfie: `/^data:image\/(jpeg|png);base64,/` and length < 900000. Element **types** inside the descriptor array are never checked.
- **Authorization:** Authenticated session only. Any signed-in user may (re)write their own enrolment row.
- **Failure mode:** Both photo UPDATEs are `.catch(() => {})` and the whole photo block is inside `try { } catch (_) { }` — "never block 2FA enrolment because a photo failed to save". The API returns `{ok:true, photoSaved}` but the page checks only `data.ok` (line 231), so the hire is told "Face 2FA enabled" whether or not a photo was stored. The descriptor is inserted via **string-assembled SQL** (`sql.raw("'" + JSON.stringify(descriptor).replace(/'/g,"''") + "'::jsonb")`), not a bound parameter. The "Skip for now" link (line 65) points at `next`, which the middleware will immediately 302 back to face setup.

---

**21. Staff-role hire enrols — and never gets a photo**

- **Actor:** User whose role is anything other than `applicant` (which includes every hire given `'editor'` at offer-send time, step 1)
- **Trigger:** 302 from step 2 to `/enroll-face`
- **System behaviour:** A two-step page: upload a government ID (descriptor extracted in-browser), then capture a live selfie; the client computes `faceapi.euclideanDistance(idDescriptor, selfieDescriptor)` and POSTs `{faceDescriptor, idDescriptor, matchDistance}` to `/api/auth/enroll-face`. **This endpoint accepts no selfie image and never writes `photo_url`.**
- **Writes:** Same `user_face_enrollments` upsert; plus `UPDATE users SET identity_verified = true, identity_verified_at = NOW(), updated_at = NOW()`; plus a best-effort `INSERT INTO identity_verifications (user_id, email, claimed_name, face_match_distance, face_match_passed=true, verdict='verified', method='self_enroll', ip_address, user_agent)`.
- **Validation:** Both descriptors must be 128-element arrays; `matchDistance` must be finite and `<= 0.55` (`FACE_MATCH_THRESHOLD`).
- **Authorization:** Authenticated session only.
- **Failure mode:** **The match distance is computed by the client and sent by the client.** The server compares a number the caller supplies against its own threshold and, on pass, sets `users.identity_verified = true` and writes a `verdict: 'verified'` audit row. A hand-crafted POST with two arbitrary 128-float arrays and `matchDistance: 0.1` produces a "verified" identity. Separately, a hire routed here never receives a profile photo — the photo path exists only in step 20.

## Known gaps / what an auditor should challenge

**Integrity**

- **G1 — The profile photo almost never gets written.** The photo is saved only when a selfie is posted to `/api/auth/enroll-face-selfie` *while* an active `hr_employees` row already exists for that `user_id`. But the middleware forces enrolment on the first protected page the account ever loads — typically at application time, months before the employee row exists — and `/portal/face-setup` then redirects away on every later visit because an active enrolment exists (`face-setup.astro:12-20`). Reaching the photo write for a real hire requires manually visiting `/portal/face-setup?force=1` *after* acceptance. Nothing in the product tells anyone to do that. Challenge: ask to see the proportion of `hr_employees` rows with a non-null `photo_url`.
- **G2 — Role decides whether a hire can ever have a verified photo.** `'applicant'` → `/portal/face-setup` (captures a photo); every other role → `/enroll-face` (no photo path exists). Since new accounts are given `'editor'` at offer-send time, the common new-hire case is routed to the page that cannot produce a photo. The admin UI at `/admin/hr/employees/[id].astro:380` nonetheless labels the image "Photo captured at face verification".
- **G3 — Identity verification trusts a client-computed number.** `/api/auth/enroll-face` accepts `matchDistance` from the request body, compares it to 0.55, and on pass sets `users.identity_verified = true` and writes `verdict: 'verified'`. There is no server-side descriptor comparison and no liveness check anywhere (the blink gate was deliberately removed, `face-setup.astro:182-184`). Any "verified identity" claim resting on this flag should be treated as unfounded.
- **G4 — The e-signature is not evidence.** `typedName` is never read; the consent checkbox has no `name` attribute and is never submitted; the stored signature is a client-generated image validated only as `data:image/*` under 500 KB; `signature_ip` is a truncated, spoofable XFF value. The UI claims permanent evidentiary recording.

**Authorization**

- **G5 — Privilege escalation on document review.** `/api/portal/onboarding/documents` with `action:'review'` requires only a non-`applicant` role, has no row scoping (`WHERE id = $id`), and operates on sequential bigserial ids. Roles middleware bans from `/admin` (`partner`, `teacher`, `technical_moderator`) can verify or reject any hire's credential. Note the mechanism: the admin guards are path-keyed to `/admin*`, so an `/api/*` route was never in their scope — this is not an `isExempt` bug.
- **G6 — Status can be silently reset.** Any `status` other than `'verified'`/`'rejected'` falls through to `'submitted'` and nulls `review_note`, which also makes the row deletable by the hire again (`removeDoc` only blocks `'verified'`). A verified credential can be laundered back to unreviewed by a crafted request.
- **G7 — Offer read bypass.** `isAdmin = user.role !== 'applicant'` skips the ownership check on `GET /portal/offer/[token]`, exposing signature image, IP and user agent to roles with no HR relationship.
- **G8 — No offer linkage on the document flow.** `/portal/onboarding` checks only that someone is signed in. A rejected applicant or an unrelated learner account can create up to five rows that appear in the HR review queue.
- **G9 — No CSRF protection.** `security: { checkOrigin: false }` globally, `sameSite 'lax'` cookies, plain `<form method="POST">` on the offer page. The JSON endpoints rely on content-type-forced preflight rather than a token or origin check.

**Durability and data model**

- **G10 — No transaction on hire.** Four independent statements. Every partial-failure combination is reachable and none is detected or reconciled.
- **G11 — Employee creation fails silently and the candidate is congratulated anyway.** `try { await transferApplicantToEmployee(offer); } catch (_) {}`; the return value is never inspected; a blank email or an `employee_code` collision (`Math.random().toString(36).slice(2,7)`, no retry) yields no employee row while the application reads `hired`.
- **G12 — Two divergent hire paths that disagree.** Offer self-acceptance → `hire-transfer.ts` (`EMP-YYYY-XXXXX`, no `department_id`, no leave balances). Admin status change → `syncHiredToEmployee` (`ERA-EMP-0001`, department resolved, `employment_type` hard-coded `'full_time'`, leave balances allocated for every active leave type). The offer page never calls `onApplicationStatusChange`.
- **G13 — Two undeclared tables on the critical path.** `hr_employees` has no schema entry, no migration and no DDL anywhere; `hr_onboarding_documents` is created by runtime `CREATE TABLE IF NOT EXISTS`; `user_face_enrollments` and `identity_verifications` have neither. `enroll-face-selfie.ts:63` runs `ALTER TABLE hr_employees ADD COLUMN IF NOT EXISTS photo_url TEXT` **inside a user request**. Ask for the authoritative production DDL — it does not exist in the repo.
- **G14 — Cap and status are code-only.** The five-document limit is a non-atomic `countDocs`-then-INSERT with no constraint; `status` is free text with no CHECK, so the `'submitted' | 'verified' | 'rejected'` union is a TypeScript fiction.
- **G15 — Photos may be stored inline.** Without `BLOB_READ_WRITE_TOKEN`, a base64 data URL up to ~900 KB is written directly into `users.photo_url` and `hr_employees.photo_url`, and rendered into admin list pages. With the token, photos land in a **publicly accessible** blob at a predictable-ish path.

**Process and observability**

- **G16 — No audit trail that anyone can see.** `reviewed_by` and `reviewed_at` are written but never selected or rendered by either UI. No audit row is written on review. `employee_id` on the documents table is always NULL and never read.
- **G17 — No notifications.** A rejected document produces no email and no in-app alert; the hire must reload the page to find out. They can then hard-DELETE the row, permanently erasing the reviewer's note. There is no soft delete and no history table.
- **G18 — Failures are invisible.** Validation and internal errors return **HTTP 200** with `{ok:false}`; `.catch(() => [])` on both list queries makes a database outage indistinguishable from an empty state on both the hire's page and the HR queue.
- **G19 — The stated BGV integration does not exist.** The `hr-onboarding.ts` header claims it "feeds the BGV education check (hr-bgv.ts)". `hr-bgv.ts` contains no reference to onboarding, and `hr_onboarding_documents` is referenced only in `hr-onboarding.ts`. Nothing downstream consumes a verified credential.
- **G20 — "Complete" means nothing.** `docs.length > 0 && rejected === 0 && verified === docs.length`. One verified document of type `other` marks a hire complete. No document type is mandatory.
- **G21 — Link verification is format-only.** The server never fetches the URL, so the instructed "Anyone with the link - Viewer" access is guidance, not enforcement; private, deleted or wrong files are accepted, and the same URL can be submitted five times.
- **G22 — Post-acceptance journey is split across two unlinked pages.** The success CTA sends the hire to `/portal/onboarding`; their next `/portal` visit redirects them to `/portal/employee/onboarding`. Neither links to the other and nothing ever clears `onboarding_status`.
- **G23 — The offer letter's own instructions are wrong for most readers.** The access-credentials block tells the hire to sign in at `/admin/login`; `middleware.ts:238-240` bounces any `applicant`-role user away from `/admin` to `/portal`. Existing-account hires keep `'applicant'` forever.
- **G24 — Admin queue does not scale.** `LIMIT 150`, no pagination, no search, no per-hire grouping, no truncation indicator, and a "total" counter that reports the page size.
- **G25 — Third-party script on an authentication surface.** Both enrolment pages load `face-api` from `cdn.jsdelivr.net` with no SRI. Compromise of that CDN compromises 2FA enrolment for every account.
- **G26 — Reason capture uses `window.prompt`.** Unstyled, unvalidatable, empty string accepted and stored as NULL, unavailable in some embedded browsers.

**Explicitly unverified — do not accept without a live-database check**

1. That `hr_onboarding_documents` exists in production (no migration, no dump, no `.dev-scripts` file references it).
2. The authoritative column set of `hr_employees` — the step-9 INSERT is the de facto contract, nothing more.
3. The table shapes of `user_face_enrollments` and `identity_verifications`, including the unique constraint on `user_id` that the `ON CONFLICT` clauses assume.
4. The offer-domain endpoint/authorization rows marked **[UNVERIFIED]** above: the audit's endpoint listing for that domain was truncated mid-entry, so only `GET /portal/offer/[token]` was independently evaluated. The corresponding step-level behaviour *was* verified.
