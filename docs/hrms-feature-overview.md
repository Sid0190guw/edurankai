## Module detail

### 1. Leave Management (HRMS)

**Purpose.** Lets an employee apply for time off against a fixed annual per-type allowance and lets a reporting manager, HR head, admin or super-admin approve or reject it, with balances derived on the fly from the request history rather than from a stored ledger.

**Status:** partial

**Features that exist**
- **Fixed leave-type catalogue (code constant, not a table)** - `LEAVE_TYPES` is a hardcoded array of exactly four entries: `casual` (Casual, allowance 12), `sick` (Sick, 12), `earned` (Earned / privilege, 15), `unpaid` (Unpaid, allowance 0). `TYPE_IDS` is a Set built from it and is the only validation of the submitted `leave_type`. There is no admin CRUD for types and no per-employee, per-grade or per-location policy. _( src/lib/hr-leave.ts:11-17 )_
- **Self-bootstrapping schema (lazy CREATE TABLE)** - `ensureLeaveSchema()` runs `CREATE TABLE IF NOT EXISTS hr_leave_request (...)` plus two indexes (`hr_leave_status` on `(status, requested_at DESC)`, `hr_leave_emp` on `(employee_id, start_date DESC)`) behind a module-level `ready` promise, so the table is created on first use rather than by a migration. On failure it resets `ready = null` and swallows the error. _( src/lib/hr-leave.ts:19-37 )_
- **Derived balance calculation (no stored balance rows)** - `getBalances(employeeId, year?)` runs one GROUP BY over `hr_leave_request` summing `days` for `status='approved'` into `used` and `status='pending'` into `pending`, filtered by `EXTRACT(YEAR FROM start_date) = year` (defaults to the current calendar year). `remaining = Math.max(0, allowance - used - pending)`; for `unpaid` (allowance 0) it returns `Infinity`. Nothing is persisted — the balance is recomputed on every page load. _( src/lib/hr-leave.ts:45-58 )_
- **Leave application with inline balance enforcement** - `applyLeave(employeeId, type, start, end, reason)` rejects unknown types, computes `days` via `daysBetween()` (inclusive calendar-day count, rejects end < start), and for any type with `allowance > 0` re-reads the balance for the year of `start` and rejects with `Only N <type> day(s) remaining this year.` if `days > remaining`. On success it INSERTs a row that defaults to `status='pending'`. _( src/lib/hr-leave.ts:39-43, 60-72 )_
- **Employee self-cancel of a pending request** - `cancelLeave(id, employeeId)` sets `status='cancelled'` with the employee id and `status='pending'` both in the WHERE clause, so an employee can only withdraw their own not-yet-decided request. The call is `.catch(() => {})` — failures are silent and the UI still shows 'Request cancelled.' _( src/lib/hr-leave.ts:83-86; src/pages/portal/employee/leave.astro:21, 89 )_
- **Approve / reject decision with role capture** - `decideLeave(id, user, decision, note)` loads the request, refuses if it is not `pending` ('Already <status>.'), then calls `approverRole(user, l.employee_id)`; a null role returns 'You are not permitted to decide this request.' On success it writes `status`, `decided_by`, `decided_by_role`, `decided_at = NOW()` and `decision_note`. _( src/lib/hr-leave.ts:88-97 )_
- **Approver permission chain (shared with payouts)** - `approverRole(user, employeeId)` returns 'super_admin', 'admin', 'hr_head' (when `user.role` merely *contains* the substring 'hr'), or 'reporting_manager' when `hr_employees.reporting_manager_user_id` or `manager_user_id` matches `user.id`; otherwise null. It is imported from the wallet module, so leave and salary withdrawals share one permission chain. _( src/lib/hr-wallet.ts:107-119, imported at src/lib/hr-leave.ts:6 )_
- **Admin approvals queue** - /admin/hr/leave loads `listLeave({})` (admin JOIN onto `hr_employees` for `full_name`, `employee_code`, `designation`, ordered pending-first, LIMIT 120), splits it into a 'Pending · N' section with inline Approve/Reject buttons plus an optional free-text `note` field, and a read-only 'Recent' section capped at 25 decided rows showing status and `decided_by_role`. POST-redirect-GET via `?msg=`. _( src/pages/admin/hr/leave/index.astro:20-61 )_
- **Admin route role gate** - The page redirects to /admin/login without a session, and to /admin unless `role === 'super_admin' || role === 'admin' || role.indexOf('hr') >= 0`. Note this is a *different* check from `approverRole` — a reporting manager passes `approverRole` but is redirected away from the page. _( src/pages/admin/hr/leave/index.astro:5-8 )_
- **Employee self-service leave page** - /portal/employee/leave resolves the `hr_employees` row from `user_id`, falling back to `work_email`/`personal_email`/`email`, and shows 'No employee profile linked' if neither matches. It renders balance cards (only types where `allowance > 0`, showing remaining / used / pending), an apply form (type, from, to, reason), and the last 60 requests with a status pill and a cancel control. Linked from the /portal/employee hub tile. _( src/pages/portal/employee/leave.astro:9-13, 62-94; src/pages/portal/employee.astro:278 )_
- **Sidebar navigation entry** - AdminLayout registers `{ id: 'hr-leave', label: 'Leave', href: '/admin/hr/leave', icon: 'calendar' }` and maps the id to the 'leave' nav group. _( src/layouts/AdminLayout.astro:91, 211 )_
- **SEPARATE, ORPHANED leave schema still wired into three screens** - A second, incompatible leave model exists and is READ but never created or written by the live module: `hr_leave_types` (`id`, `name`, `code`, `is_paid`, `days_per_year`, `is_active`), `hr_leave_balances` (`employee_id`, `leave_type_id`, `year`, `allocated`, `used`, `carried_forward`, unique on `(employee_id, leave_type_id, year)`) and `hr_leave_requests` — PLURAL, with `leave_type_id` and `applied_at`. `syncHiredToEmployee` allocates balance rows from `hr_leave_types` on hire; the employee-detail Leave tab renders `allocated + carried_forward - used` with a usage bar and the last 5 `hr_leave_requests`; both dashboards count `SELECT COUNT(*) FROM hr_leave_requests WHERE status = 'pending'`. Nothing submitted through /portal/employee/leave lands in these tables. _( src/lib/hr/sync.ts:85-93; src/pages/admin/hr/employees/[id].astro:307-314, 341-348, 880-909; src/pages/admin/index.astro:33; src/pages/admin/hr/index.astro:16 )_

**Tables**
- `hr_leave_request (SINGULAR — the only leave table with DDL in the repo, created lazily by ensureLeaveSchema at src/lib/hr-leave.ts:24-31): id UUID PK DEFAULT gen_random_uuid(), employee_id UUID NOT NULL, leave_type TEXT NOT NULL, start_date DATE NOT NULL, end_date DATE NOT NULL, days INT NOT NULL, reason TEXT, status TEXT NOT NULL DEFAULT 'pending' (comment documents pending|approved|rejected|cancelled), requested_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), decided_by UUID, decided_by_role TEXT, decided_at TIMESTAMPTZ, decision_note TEXT. Indexes: hr_leave_status (status, requested_at DESC), hr_leave_emp (employee_id, start_date DESC). No FK to hr_employees, no CHECK on status, no uniqueness/overlap constraint.`
- `hr_leave_requests (PLURAL — different table, NO DDL anywhere in the repo). Columns proven only by the queries that read it: employee_id, leave_type_id (FK-style join to hr_leave_types.id), status, applied_at, plus `lr.*`. Read at src/pages/admin/hr/employees/[id].astro:341-348 and counted at src/pages/admin/index.astro:33 and src/pages/admin/hr/index.astro:16. Never inserted into by any code in this repo. Full column set UNVERIFIED.`
- `hr_leave_types (NO DDL in the repo). Columns proven by reads/joins: id, name, code, is_paid, days_per_year, is_active. Read at src/lib/hr/sync.ts:85 and src/pages/admin/hr/employees/[id].astro:308-310, 342-344. Never written by any code in this repo — the catalogue must be seeded out of band.`
- `hr_leave_balances (NO DDL in the repo). Columns proven by the INSERT and the SELECT: employee_id, leave_type_id, year, allocated, used, carried_forward. The `ON CONFLICT (employee_id, leave_type_id, year) DO NOTHING` at src/lib/hr/sync.ts:91 implies a unique constraint on that triple (inferred, not verified). Written only by syncHiredToEmployee; `used` and `carried_forward` are never updated anywhere in the repo.`
- `hr_employees — joined for display only (full_name, employee_code, designation at src/lib/hr-leave.ts:77-78) and for the approver check (reporting_manager_user_id / manager_user_id at src/lib/hr-wallet.ts:114-116, tried in that order because the column name varies).`
- `hr_attendance — NOT written by this module. It carries an 'on_leave' status value (src/pages/admin/hr/attendance/index.astro:192, 200, 228, 455, 530) but nothing in hr-leave.ts creates or updates an attendance row when leave is approved.`

**Admin surface.** Single route: **/admin/hr/leave** (src/pages/admin/hr/leave/index.astro), reachable from the AdminLayout sidebar as 'Leave' (icon 'calendar'). Gate: signed-in and `role === 'super_admin' || role === 'admin' || role.indexOf('hr') >= 0`, otherwise redirected to /admin. HR sees one global queue — 'Pending · N' with every organisation-wide pending request (LIMIT 120, pending sorted first), each row showing employee name, capitalised leave type, day count, employee code, formatted date range (en-IN) and reason, with an inline form carrying an optional free-text 'note' and two submit buttons (Approve / Reject). Below it a read-only 'Recent' list of up to 25 decided requests at 75% opacity showing the colour-coded status and `decided_by_role`. Decisions are a plain form POST handled in the same file (`action=decide`), re-authorised server-side by `decideLeave` -> `approverRole`, then redirected to `?msg=`. The queue is NOT scoped to the viewer's reportees, has no filter/search/pagination UI (the `opts.status` argument `listLeave` supports is never passed), and there is no bulk action, no allocation/adjustment screen, no leave-type editor and no holiday calendar. Separately, **/admin/hr/employees/[id]** has a 'LEAVE' tab (panel-3) that reads the *other*, orphaned schema — per-type allocated/used/carried_forward with a usage bar and the 5 most recent `hr_leave_requests` — so it will read 'No leave balances set for <year>' for anyone whose balances were never seeded, regardless of what they applied for.

**Employee surface.** **/portal/employee/leave** (src/pages/portal/employee/leave.astro), a mobile-width self-contained page (max-width 560px, paper/ink palette, no shared layout) linked from the /portal/employee hub tile. It resolves the employee by `hr_employees.user_id = user.id`, falling back to `work_email`/`personal_email`/`email`, and shows 'No employee profile linked' if neither hits. The employee can: (1) see 'Leave balance · <current year>' cards for the three types with an allowance — big remaining number plus 'N used, M pending' (the `unpaid` type is filtered out of this grid by `b.allowance > 0`); (2) apply, choosing a type (unpaid is labelled 'Unpaid (no allowance)'), a from/to date pair and a free-text reason, with the page stating honestly 'Counts calendar days between the dates (inclusive)'; (3) review their last 60 requests with a colour pill (pending amber / approved green / rejected red / cancelled grey); (4) cancel any still-pending request with the '×' control. Errors from `applyLeave` (invalid range, insufficient balance) are surfaced verbatim through the `?msg=` flash. The employee cannot see who approved it, the approver's note, or any holiday/team calendar.

**Not built / half-built**
- NO HOLIDAY CALENDAR EXISTS ANYWHERE. There is no holiday table, no calendar UI and no weekend logic. 'holiday' appears only as one selectable *attendance status* value in src/pages/admin/hr/attendance/index.astro (STATUS_BG:193, STATUS_TEXT:201, icon:229, dropdowns:454 and 531). `daysBetween` (src/lib/hr-leave.ts:39-43) is a raw inclusive calendar-day count, so a Mon-Fri-spanning-a-weekend request or a request over a public holiday debits the employee's balance for those days.
- TWO INCOMPATIBLE LEAVE SCHEMAS ARE LIVE AT ONCE. The working flow writes `hr_leave_request` (singular, TEXT `leave_type`); the employee-profile Leave tab and BOTH dashboard counters read `hr_leave_requests`/`hr_leave_types`/`hr_leave_balances` (plural, `leave_type_id`). Consequence: the 'pending leave' KPI on /admin and /admin/hr (src/pages/admin/index.astro:33, src/pages/admin/hr/index.astro:16) will never reflect a request submitted through /portal/employee/leave, and neither will the employee's own profile page.
- NO DDL IN THE REPO for hr_leave_types, hr_leave_balances or hr_leave_requests. They are only ever read (and, for balances, inserted into by src/lib/hr/sync.ts:85-93). Their real column sets and constraints cannot be established from this codebase. `hr_leave_types` is never seeded by any code here, so on a fresh database `syncHiredToEmployee` allocates zero rows and the Leave tab is permanently empty.
- NO ACCRUAL ENGINE. The word 'accrual' does not exist in the module. Allowance is a hardcoded whole-year number granted in full on 1 January; there is no monthly/pro-rata accrual, no mid-year joiner proration, no probation hold, no carry-forward job, no encashment and no year-end rollover. `carried_forward` is read at src/pages/admin/hr/employees/[id].astro:886 but is never written by any code in the repo.
- NO HALF-DAY OR HOURLY LEAVE. `days` is `INT` and `daysBetween` returns whole days, so 0.5-day requests are impossible even though the attendance module has a `half_day` status.
- NO OVERLAP OR DUPLICATE DETECTION. Nothing checks whether a new request's date range collides with an existing pending or approved one; an employee can stack overlapping requests until the aggregate balance check trips.
- THE REPORTING-MANAGER APPROVAL PATH HAS NO UI. `approverRole` can return 'reporting_manager' (src/lib/hr-wallet.ts:116), but the only page that calls `decideLeave` redirects anyone who is not admin/super_admin/'hr'-substring away (src/pages/admin/hr/leave/index.astro:8). A plain reporting manager therefore has the permission but no screen. Conversely the HR queue is unscoped — every HR user sees and can decide every employee's request.
- NO API LAYER. There are no leave endpoints under src/pages/api; both surfaces are Astro form POSTs only, so nothing external (mobile app, integration) can read balances or file a request.
- NO NOTIFICATIONS. No email, in-app alert or thread is raised when leave is applied for, approved, rejected or cancelled — the employee only learns the outcome by revisiting the page.
- NO DOWNSTREAM INTEGRATION. Approving leave does not write an `on_leave` row into `hr_attendance`, and `unpaid` leave does not feed any loss-of-pay deduction into payroll (`hr_payroll_runs`). The three modules are entirely decoupled.
- SILENT FAILURE MODE. `safe()` (src/lib/hr-leave.ts:9) swallows every query error and returns `[]`, and `ensureLeaveSchema` catches and resets on failure. A database outage or a missing table renders as 'No leave requests waiting.' / 'No leave requests yet.' rather than an error — an approver cannot distinguish an empty queue from a broken one.
- YEAR-BOUNDARY BUG. Balances key off `EXTRACT(YEAR FROM start_date)` only (src/lib/hr-leave.ts:52) and applyLeave checks against the year of `start` (line 67). A 28 Dec - 4 Jan request debits all 8 days from the *starting* year's allowance and none from the new year's.
- NO AUDIT TRAIL BEYOND THE ROW ITSELF. `decided_by`/`decided_by_role`/`decided_at`/`decision_note` are overwritten in place; there is no history table, so a reversed or re-decided request leaves no trace. The status is also not constrained by a CHECK, and there is no FK from `hr_leave_request.employee_id` to `hr_employees.id`.
- APPROVER LOOKUP USES STRING-INTERPOLATED SQL. `approverRole` builds its query with `sql.raw` and strips quotes by hand (`employeeId.replace(/'/g, "")`, src/lib/hr-wallet.ts:115) instead of binding parameters. It is reached from leave decisions with an id that came from the request row, but the pattern is fragile and worth replacing with a bound query.
- MINOR: the 'hr' role gate is a substring test (`role.indexOf('hr') >= 0`) in both the page and `approverRole`, so any future role whose name merely contains 'hr' silently gains organisation-wide leave approval rights.

### 2. Attendance & Time Tracking (HRMS)

**Purpose.** Records who worked when — self-service clock in/out/break from the employee portal (with GPS and an optional selfie), plus an HR-side day view where attendance can be marked, corrected, bulk-set and read alongside each employee's hourly time log and written daily report.

**Status:** partial

**Features that exist**
- **Employee clock in / out / break / resume** - Four event types ('clock_in','clock_out','break_start','break_end') POSTed as action=clock_event. Every event inserts a row into hr_clock_events with employee_id, event_type, lat, lon, accuracy, location_name, work_mode, note, face_photo, ip_address (Astro.clientAddress) and device_info (user-agent, first 200 chars). Current state is derived from the LAST event of the day: clock_in/break_end => clocked_in, break_start => on_break, clock_out => clocked_out. _( src/pages/portal/employee.astro lines 39-88 (handler), 180-206 (state derivation), 328-400 (buttons) )_
- **Clock event writes through to the daily attendance row** - clock_in does INSERT INTO hr_attendance (employee_id, date, clock_in, status, work_mode, ip_address) VALUES (..., NOW(), 'present', ...) ON CONFLICT (employee_id, date) DO UPDATE SET clock_in = NOW(), status='present'. clock_out does UPDATE hr_attendance SET clock_out = NOW(), work_hours = EXTRACT(EPOCH FROM (NOW() - clock_in))/3600. Both are wrapped in .catch(() => {}) so a failure is silent. _( src/pages/portal/employee.astro lines 59-71 )_
- **Geolocation capture on every clock action** - Browser navigator.geolocation.getCurrentPosition with enableHighAccuracy:true, timeout 12000, maximumAge 30000. Position is prefetched on page load and shown in a chip ('Location ready · accurate to ±Nm'). On submit, lat/lon/accuracy are written into hidden fields and location_name is composed client-side as '±Nm · lat, lon' (5 decimals). A 13s hard fallback submits anyway; if geolocation is denied, location_name is set to the literal string 'location unavailable' and the clock event still proceeds. _( src/pages/portal/employee.astro lines 531-598 (prefetch, fill, finishClock) )_
- **Face photo (selfie) on clock actions** - captureFace() opens an in-page overlay with the front camera (facingMode:'user'), the user presses 'Capture & continue', a 240px-wide JPEG at quality 0.6 is drawn to canvas and stored as a base64 data URI in a hidden face_photo field (server-truncated to 300,000 chars). Explicitly skippable ('Skip camera') and auto-skipped if getUserMedia is missing or denied — the code comment states it never blocks someone from marking attendance. _( src/pages/portal/employee.astro lines 545-604 (setFace, captureFace, submitClock); server side line 47 )_
- **Face check-in strip in the HR day view** - Queries the day's hr_clock_events WHERE face_photo IS NOT NULL (LIMIT 60) and renders a horizontally scrolling row of 76px thumbnails, each mirrored (transform:scaleX(-1)), with an IN/OUT/BRK/RES badge, first name and time. Labelled '{n} · advisory — verify visually'. The whole query is in a try/catch because the face_photo column self-bootstraps. _( src/pages/admin/hr/attendance/index.astro lines 131-143 (query), 286-306 (render) )_
- **Manual mark & edit attendance (single employee)** - A bottom-sheet modal (id=markModal) posting action=mark. Fields: employee, date, status (present/absent/wfh/half_day/on_leave/holiday), work mode (remote/office/hybrid), clock in, clock out, notes. Upserts on (employee_id, date). Work hours are computed server-side; if clock_out <= clock_in it assumes an overnight shift and rolls clock_out to the next calendar day before computing (the fix for the '-7.50h' bug named in the comment). editAttendance(btn) reuses the same modal, prefilled from data-* attributes. _( src/pages/admin/hr/attendance/index.astro lines 18-63 (handler), 473-499 (editAttendance), 501-558 (modal) )_
- **Delete an attendance entry** - action=delete runs DELETE FROM hr_attendance WHERE employee_id = ... AND date = ... behind a JS confirm(). No soft delete, no audit record. _( src/pages/admin/hr/attendance/index.astro lines 64-68, 356-361 )_
- **Bulk mark all active employees** - action=bulk_mark selects every hr_employees row WHERE is_active = true and loops one INSERT per employee with ON CONFLICT (employee_id, date) DO NOTHING, so it will not overwrite existing rows. Status choices here are present/absent/wfh/holiday/on_leave; work mode remote/office/hybrid. Reports 'Marked N employees as X for DATE'. _( src/pages/admin/hr/attendance/index.astro lines 69-86 (handler), 437-470 (Bulk Mark tab) )_
- **Daily tab: per-employee attendance cards** - Joins hr_attendance to hr_employees, LEFT JOINs hr_daily_reports on (employee_id, report_date) and adds two correlated subqueries for SUM(hours) and COUNT(*) from hr_time_logs. Each card shows a status glyph (inline SVG, no emoji), clock in → clock out in Asia/Kolkata, '(prev)'/'(next day)' markers when the shift crosses midnight, 'Xh clocked', 'Yh logged (N)', work mode, the expanded hourly time log, and the daily diary (work done / progress / blockers) or the warning 'No daily report submitted for this day.' _( src/pages/admin/hr/attendance/index.astro lines 110-126, 284-393 )_
- **Not-yet-marked warning list** - SELECT of active employees whose id is NOT IN (SELECT employee_id FROM hr_attendance WHERE date = ?) rendered as an amber banner with the first 8 first names plus a '+N more' pill. _( src/pages/admin/hr/attendance/index.astro lines 146-152, 307-317 )_
- **Day stats strip and monthly per-employee summary** - Day strip counts present/absent/wfh/half/on_leave for the selected date. The Monthly tab takes one employee and returns COUNT(*) FILTER by status, ROUND(AVG(work_hours),1) and SUM(overtime_hours) for DATE_TRUNC('month', a.date) = DATE_TRUNC('month', NOW()) — note the rendered tiles use present/absent/wfh/half/leave days and avg hours; total_overtime is selected but never displayed. _( src/pages/admin/hr/attendance/index.astro lines 154-183, 256-270, 396-434 )_
- **Hourly time log (period tracking)** - action=add_time_log validates HH:MM start/end with /^(\d{1,2}):(\d{2})$/, requires end after start and a task of at least 3 characters, computes hours = (end-start)/60 rounded to 2dp, inserts into hr_time_logs. action=delete_time_log deletes by id scoped to the signed-in employee. The portal card shows today's blocks and a running 'Xh logged' total. _( src/pages/portal/employee.astro lines 108-131 (handlers), 134-144 (fetch), 422-450 (UI) )_
- **Daily work report / diary** - action=daily_report requires work_done of at least 10 characters, plus optional progress and blockers. Upserts on (employee_id, report_date) with updated_at = NOW(), so the employee can edit and resubmit; the card shows a 'submitted' marker when a report exists for today. _( src/pages/portal/employee.astro lines 90-106, 155-164, 452-471 )_
- **Clock-out task wrap-up** - The Clock Out form also carries task, task_status (done/pending/blocked) and a task_percent range slider (0-100, step 5) plus optional task_details. On clock_out these are written to hr_task_log — a table created lazily by a CREATE TABLE IF NOT EXISTS executed inside the request handler. The employee sees them back as 'Today's session summaries'. _( src/pages/portal/employee.astro lines 72-79 (write), 146-153 (read), 352-375 and 404-420 (UI) )_
- **Overnight-shift self-heal on page load** - Every GET of the admin attendance page runs UPDATE hr_attendance SET clock_out = clock_out + INTERVAL '1 day', work_hours = ROUND(EXTRACT(EPOCH FROM ((clock_out + INTERVAL '1 day') - clock_in))/3600.0, 2) WHERE clock_out < clock_in. Described in-code as idempotent and one-shot; errors are swallowed by an empty catch. _( src/pages/admin/hr/attendance/index.astro lines 97-107 )_
- **Employee-to-user linking fallback** - The portal first looks up hr_employees by user_id; if none, it matches on work_email, personal_email or email against the session user's email and then back-fills hr_employees.user_id. If nothing matches, the page shows 'No employee profile found — ask HR to link your account.' _( src/pages/portal/employee.astro lines 15-33, 247-253 )_
- **Attendance feeds payroll day counts** - Payroll run generation counts COUNT(*) FILTER (WHERE status='present' OR status='wfh') as days_worked, status='on_leave' as days_leave and status='absent' as days_absent for the month, and writes them into hr_payslips.days_worked / days_leave / days_absent. Note it counts days by status only — work_hours and hr_time_logs hours are not used. _( src/pages/admin/hr/payroll/index.astro lines 86-112 )_
- **Attendance on other admin screens** - /admin/hr shows a 'Marked Today' KPI (COUNT(*) FROM hr_attendance WHERE date = CURRENT_DATE) linking to the attendance page. /admin/hr/employees/[id] lists the last 10 hr_attendance rows for that employee. /admin/users/[id] derives a live status chip ('Working' / 'On break' / 'Clocked out' / 'Not clocked in') from today's hr_clock_events. _( src/pages/admin/hr/index.astro line 17 and 128; src/pages/admin/hr/employees/[id].astro lines 333-338 and 932-946; src/pages/admin/users/[id].astro lines 26-34 and 72-74 )_
- **RBAC gating on the admin surface** - src/middleware.ts maps '/admin/hr/attendance' to the permission section key 'attendance'; a user whose custom role does not grant view on that key is 302'd to /admin?denied=attendance. The page itself additionally redirects a missing session to /admin/login and role 'applicant' to /portal. Both /admin/* and /portal/* also sit behind the face-enrolment 2FA gate. _( src/middleware.ts lines 21 and 265-273; src/pages/admin/hr/attendance/index.astro lines 6-8 )_

**Tables**
- `hr_attendance — employee_id, date, clock_in, clock_out, work_hours, overtime_hours, status ('present'|'absent'|'wfh'|'half_day'|'on_leave'|'holiday'), work_mode, notes, ip_address; unique constraint on (employee_id, date) (relied on by every ON CONFLICT). **Purpose.** Monthly India-centric payroll for active employees: HR sets a per-employee salary structure, generates a month/year payroll run that auto-computes earnings, statutory deductions and net pay into per-employee payslip rows, walks the run through draft to approved to paid, and renders a printable HTML payslip.

### 4. HR Wallet / Payouts

**Purpose.** A double-entry employee wallet ledger where HR credits salary/bonus/reimbursement into an employee wallet, the employee registers a bank account and requests a withdrawal, an approver (reporting manager, HR head, admin or super-admin) approves it, and HR releases the money either automatically via RazorpayX Payouts or as a manually-recorded settlement.

**Status:** partial

**Features that exist**
- **Self-bootstrapping schema** - `ensureWalletSchema()` runs `CREATE TABLE IF NOT EXISTS` for `hr_wallet_txn`, `hr_bank_account`, `hr_withdrawal` plus two indexes (`hr_wallet_txn_emp`, `hr_withdrawal_status`) on first use, memoised in a module-level `ready` promise. Every public function awaits it. On error it swallows the exception and resets `ready = null` so the next call retries. _( src/lib/hr-wallet.ts:13-47 )_
- **Balance computation (balance / pending / available)** - `getBalance(employeeId)` returns `{ balance, currency, pending, available }`. `balance` is `SUM(CASE WHEN direction='credit' THEN amount ELSE -amount END)` over `hr_wallet_txn`; `pending` is `SUM(amount)` of that employee's `hr_withdrawal` rows in status `'pending'` or `'approved'`; `available = Math.max(0, balance - pending)`. Currency is hardcoded to the string `'INR'` in the return object. _( src/lib/hr-wallet.ts:49-58 )_
- **Credit a wallet (HR-initiated)** - `credit(employeeId, amount, kind, note, createdBy, ref?)` inserts a `direction='credit'` row into `hr_wallet_txn`. Silently no-ops if `amount` is not > 0. `kind` is free text; the admin form offers `salary`, `bonus`, `reimbursement`, `adjustment`. _( src/lib/hr-wallet.ts:63-68; form handler src/pages/admin/hr/wallet/index.astro:21 )_
- **Transaction ledger listing** - `listTxns(employeeId, limit = 30)` selects `direction, amount, currency, kind, note, created_at` newest-first. The employee page requests 20 and renders credits green with a `+` and debits red with a `−`. _( src/lib/hr-wallet.ts:59-62; rendered src/pages/portal/employee/wallet.astro:132-140 )_
- **Bank account registration with masking** - `addBankAccount()` demotes all existing rows to `is_primary = false`, then inserts the new one as primary. Inputs are truncated (`holder` 120, `account_number` 30 with whitespace stripped, `ifsc` 15 uppercased, `bank_name` 120, `upi_id` 80). `listBankAccounts()` adds a computed `account_masked` field from the local `mask()` helper (`'••••' + last 4`). _( src/lib/hr-wallet.ts:71-82 )_
- **Withdrawal request with balance guard** - `requestWithdrawal(employeeId, amount, bankAccountId, note)` re-reads the balance and rejects amounts <= 0 ('Enter an amount.') or above `bal.available` ('Amount exceeds your available balance (₹…)'), otherwise inserts a `hr_withdrawal` row defaulting to `status = 'pending'`. _( src/lib/hr-wallet.ts:85-92 )_
- **Withdrawal listings (two modes)** - `listWithdrawals({ employeeId?, status? })` — with `employeeId` it returns that employee's own rows (LIMIT 50); without it, the admin variant LEFT JOINs `hr_employees` (`full_name`, `employee_code`, `designation`) and `hr_bank_account` (`holder`, `account_number`, `ifsc`, `upi_id`), ordered `(w.status='pending') DESC, w.requested_at DESC` LIMIT 100. _( src/lib/hr-wallet.ts:93-103 )_
- **Approver permission chain (shared with leave)** - `approverRole(user, employeeId)` returns `'super_admin'`, `'admin'`, `'hr_head'` (matched by `role.indexOf('hr') >= 0`, a substring test), or `'reporting_manager'` if `hr_employees` links the user via `reporting_manager_user_id` or `manager_user_id`. Returns `null` otherwise. This same function is imported by the leave module as its approval chain. _( src/lib/hr-wallet.ts:107-119; reused in src/lib/hr-leave.ts:6 )_
- **Approve / reject a withdrawal** - `decideWithdrawal(id, user, 'approved'|'rejected', note)` refuses if the row is missing or not `pending` ('Already ' + status), then stamps `status`, `decided_by`, `decided_by_role`, `decided_at = NOW()`, `decision_note`. _( src/lib/hr-wallet.ts:121-130 )_
- **RazorpayX payout release + wallet debit** - `payWithdrawal(id, user, manualRef?)` requires `status = 'approved'` and a role of `admin`, `super_admin` or `hr_head` (reporting managers are explicitly excluded from releasing money). If no `manualRef` is given and `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, `RAZORPAYX_ACCOUNT_NUMBER` and a stored `account_number` all exist, it POSTs to `https://api.razorpay.com/v1/payouts` with HTTP Basic auth (`Buffer.from(KEY + ':' + SEC).toString('base64')`), body `{ account_number: ACC, amount: Math.round(amount * 100), currency, mode: 'IMPS', purpose: 'payout', fund_account: { account_type: 'bank_account', bank_account: { name, ifsc, account_number } }, queue_if_low_balance: true, narration: 'Salary withdrawal' }`. On non-OK it returns 'Razorpay: ' + description and changes nothing. On success `ref = j.id`, `paidVia = 'razorpayx'`. Either path then sets `status='paid'`, `payout_ref`, `paid_at = NOW()` and inserts the offsetting `direction='debit'`, `kind='withdrawal'` ledger row with note `'Withdrawal via ' + paidVia`. _( src/lib/hr-wallet.ts:134-162 )_
- **Payout-channel status banner** - The admin page computes `rzxReady = !!process.env.RAZORPAYX_ACCOUNT_NUMBER` and shows either 'RazorpayX Payouts — automatic on release' or 'manual settlement' with the instruction to set `RAZORPAYX_ACCOUNT_NUMBER`. When not ready, the release form swaps in a `manual_ref` input ('UTR / reference') and the button reads 'Mark paid' instead of 'Release via RazorpayX'. _( src/pages/admin/hr/wallet/index.astro:34, 46-48, 89-90 )_
- **Admin triage board (pending / approved / recent)** - `listWithdrawals({})` is bucketed in-page into `pending`, `approved`, and `settled` (`paid` | `rejected` | `failed`, sliced to 20). All three sections render with counts in the heading. Bank details in the pending row are masked for display (`'••••' + last 4` plus IFSC). _( src/pages/admin/hr/wallet/index.astro:26-29, 64-104 )_
- **Employee profile auto-link** - The employee wallet page resolves `hr_employees` by `user_id`, falling back to matching `work_email`, `personal_email` or `email` against the session email; if a row is found without a `user_id` it back-fills `user_id` so future loads hit the fast path. If nothing matches it renders 'No employee profile linked'. _( src/pages/portal/employee/wallet.astro:11-15, 79 )_

**Tables**
- `hr_wallet_txn — id UUID PK (gen_random_uuid), employee_id UUID NOT NULL, direction TEXT ('credit'|'debit'), amount NUMERIC(14,2), currency TEXT DEFAULT 'INR', kind TEXT DEFAULT 'adjustment' (salary|bonus|reimbursement|withdrawal|adjustment), ref TEXT, note TEXT, created_by UUID, created_at TIMESTAMPTZ DEFAULT NOW(); index hr_wallet_txn_emp (employee_id, created_at DESC)`
- `hr_bank_account — id UUID PK, employee_id UUID NOT NULL, holder TEXT, account_number TEXT, ifsc TEXT DEFAULT '', bank_name TEXT DEFAULT '', upi_id TEXT DEFAULT '', is_primary BOOLEAN DEFAULT true, verified BOOLEAN DEFAULT false, rzp_fund_account_id TEXT, created_at TIMESTAMPTZ DEFAULT NOW()`
- `hr_withdrawal — id UUID PK, employee_id UUID NOT NULL, amount NUMERIC(14,2), currency TEXT DEFAULT 'INR', bank_account_id UUID, method TEXT DEFAULT 'bank', status TEXT DEFAULT 'pending' (pending|approved|rejected|paid|failed), note TEXT, requested_at TIMESTAMPTZ DEFAULT NOW(), decided_by UUID, decided_by_role TEXT, decided_at TIMESTAMPTZ, decision_note TEXT, payout_ref TEXT, paid_at TIMESTAMPTZ; index hr_withdrawal_status (status, requested_at DESC)`

**Admin surface.** `/admin/hr/wallet` (src/pages/admin/hr/wallet/index.astro), rendered in `AdminLayout` with title "Wallet & payouts". Access gate: redirect to `/admin/login` if no session, then to `/admin` unless role is `super_admin`, `admin`, or contains the substring `hr`. Three POST actions, all PRG (redirect back with `?msg=`): `disburse` (credit an active employee's wallet — employee dropdown from `SELECT id, full_name, employee_code, designation FROM hr_employees WHERE is_active = true ORDER BY full_name ASC LIMIT 500`, amount, kind, note), `decide` (Approve / Reject with an optional reason), and `pay` (release — 'Release via RazorpayX' when configured, else 'Mark paid' with a UTR/reference field). Sections shown: payout-channel banner, "Credit a wallet", "Pending approval · N", "Approved · ready to pay · N", and "Recent" (last 20 settled, dimmed, with `payout_ref`).

**Employee surface.** `/portal/employee/wallet` (src/pages/portal/employee/wallet.astro), a standalone mobile-first page (max-width 560px, own `<style>`, not using a shared layout), linked from the `/portal/employee` "My HR" dashboard tile. Requires a session (else redirect to `/portal/login`). Shows: a dark balance card with **available balance** in large type plus "In wallet" and "Pending withdrawal" sub-figures; a "Withdraw to bank" form (amount capped at `bal.available` via the `max` attribute, target account select, optional note) that is replaced by "Connect a bank account below before you can withdraw." when no bank exists; "Your withdrawals" with colour-coded status pills (pending amber, approved blue, paid green, rejected/failed red); "Bank accounts" with masked numbers, a `primary` pill, and an expandable add/change form (holder, account number, IFSC, bank name, optional UPI id — requires holder + account number + either IFSC or UPI); and a "Transactions" ledger of the last 20 entries. The employee sees no raw approver identity, only the status.

**Not built / half-built**
- No payout status reconciliation. `payWithdrawal` marks the row `'paid'` immediately after Razorpay returns a payout id, but a RazorpayX payout is asynchronous (queued/processing/reversed). There is no webhook handler, no polling job, and nothing anywhere in the codebase ever writes the `'failed'` status that the `hr_withdrawal` status comment and the admin `settled` filter both anticipate.
- No idempotency on the payout call. The POST to `https://api.razorpay.com/v1/payouts` sends no idempotency key and the status is only flipped after the call returns; a timeout followed by a retry of the 'pay' action can issue a duplicate bank transfer.
- `hr_bank_account.rzp_fund_account_id` is created in the schema but never written or read — `payWithdrawal` inlines a fresh `fund_account` object on every payout instead of reusing a stored RazorpayX fund account.
- `hr_bank_account.verified` defaults to `false`, is selected by `listBankAccounts`, and is never set to `true` by any code. There is no penny-drop / name-match verification flow, and `payWithdrawal` does not check it before paying.
- UPI payouts are not implemented. `hr_withdrawal.method` defaults to `'bank'` and is never set to anything else; `payWithdrawal` hardcodes `mode: 'IMPS'` and `account_type: 'bank_account'`, and its guard requires `w.account_number`, so a UPI-only account silently falls through to the manual path even though both UI surfaces display `upi_id` as a payout target.
- Payroll is not wired to the wallet. `/admin/hr/payroll` writes `hr_payroll_runs` / `hr_payslips` and never imports `hr-wallet` or calls `credit()`; salary must be re-entered by hand in the 'Credit a wallet' form, with no link back to the payslip that justified it.
- No `debit()` export. The only debit path is the INSERT inside `payWithdrawal`, so recoveries, advances or reversals cannot be recorded without raw SQL.
- No reversal / cancellation. An employee cannot cancel their own pending request, a rejected request cannot be reopened, and a wrong `credit()` cannot be undone through any UI.
- Race condition on available balance. `requestWithdrawal` reads `getBalance` and then INSERTs in two separate statements with no transaction, row lock, or DB-level constraint, so two concurrent requests can each pass the `amount > bal.available` check and jointly overdraw the wallet.
- `approverRole` builds its reporting-manager query with `sql.raw` and string concatenation (src/lib/hr-wallet.ts:115), sanitising only by stripping single quotes from `employeeId` and `user.id`. It is raw SQL interpolation of request-derived values rather than a bound parameter, and it silently swallows errors via `safe()` so a malformed column name is indistinguishable from 'not the manager'.
- The HR role gate is a substring test (`role.indexOf('hr') >= 0`) in both `approverRole` and the admin page's `canView`, so any role string that merely contains 'hr' is granted HR-head powers including payout release.
- `/admin/hr/wallet` is not linked from any navigation. It is rendered with `activeNav=""` and no `href` to it exists anywhere in `src/` other than its own redirect — the page is reachable only by typing the URL.
- No notifications. Neither a wallet credit, an approval, a rejection, nor a completed payout triggers anything from `src/lib/notify.ts` or `src/lib/push.ts`; the employee only learns of a decision by revisiting the page.
- No audit trail beyond the row itself. `hr_wallet_txn.created_by` and `hr_withdrawal.decided_by` are stored, but there is no append-only audit log, and `credit()` records no approval or maker-checker step — a single HR user can mint wallet balance unilaterally.
- No dedicated tests. There is no `hr-wallet.test.ts` (unlike `src/lib/hr-onboarding.test.ts`), so the balance arithmetic, the available-balance guard and the approver chain are unverified.
- `getBalance` hardcodes `currency: 'INR'` and sums across all rows regardless of the per-row `currency` column, so a non-INR `hr_wallet_txn` row would be added to the INR total.
- No pagination or filtering in the admin view — `listWithdrawals({})` is capped at LIMIT 100 and the page ignores the function's `status` option, filtering client-side in JS instead.
- No CSRF protection on any of the POST forms on either surface, and the admin 'Credit a wallet' form has no server-side upper bound on the amount.
- `ensureWalletSchema` swallows all DDL errors and resets `ready = null`; if the tables cannot be created, `safe()` returns `[]` everywhere and both pages render as an empty, healthy-looking wallet with a ₹0.00 balance rather than surfacing the failure.
- The admin listing query selects the raw `b.account_number AS bank_acc` (src/lib/hr-wallet.ts:97) even though the page only ever renders a masked form of it — the unmasked value is fetched into the page context unnecessarily. This is flagged in docs/hrms-audit-scenario.md, though that write-up overstates it as rendering full bank details.

### 5. Worker Classification

**Purpose.** A compliance register that records, for every active person the group engages, what they legally are (permanent employee, fixed-term, intern, independent contractor, EOR-employed, consultant, volunteer), which country they work in, and through what channel they are paid — so that misclassification, which the module's own header comment calls "the single biggest compliance risk in the lifecycle", is visible and reviewable rather than implicit.

**Status:** partial

**Features that exist**
- **Self-bootstrapping classification schema** - `ensureClassificationSchema()` runs idempotent DDL on first use, memoised in a module-level `ready` promise: adds `classification VARCHAR(40) DEFAULT 'permanent'`, `country_of_work VARCHAR(80) DEFAULT 'IN'`, `engaged_via VARCHAR(40)`, `classification_reviewed_at TIMESTAMPTZ`, `classification_reviewed_by UUID` to `hr_employees`, plus index `hre_classification_idx ON hr_employees(classification, country_of_work)`. No separate migration file exists. Note the whole block is wrapped in `try { } catch (_) {}` — a DDL failure is swallowed silently and the page will then fail on the SELECT. _( src/lib/hr-classification.ts:11-24 )_
- **Seven-type classification taxonomy with a fixed risk label** - `CLASSIFICATIONS` is a hardcoded const object of 7 engagement types, each with `label`, `risk` and `description`: `permanent` (low), `fixed_term` (low), `intern` (low, "Stipend-paid intern. Time-bounded. Convertible to FT."), `contractor` (high, "Genuine contractor: invoices, controls own hours and tools, paid for outcomes. Wrong classification = serious legal risk."), `eor` (low, EOR partner employs them in their country), `consultant` (medium), `volunteer` (high, "Unpaid contributor. Use sparingly; check local labour law."). The risk level is a static property of the type — it is not computed from anything about the person. _( src/lib/hr-classification.ts:26-34 )_
- **Engagement-channel taxonomy (how we pay them)** - `ENGAGEMENT_VIAS`, 6 hardcoded options separating *what they are* from *how they are engaged*: `direct_payroll`, `eor_partner`, `staffing_agency`, `intern_programme`, `contractor_direct`, `marketplace`. Stored in `hr_employees.engaged_via`; blank is allowed (the select has an empty first option). _( src/lib/hr-classification.ts:36-43 )_
- **Country-of-work register (15 codes)** - `COUNTRY_CODES` — a fixed list of 15 entries (`IN, US, GB, DE, FR, SG, AE, AU, CA, NL, CH, JP, KR, IL, OTHER`) used both to set `hr_employees.country_of_work` and to render the by-country headcount. This is what makes the register a permanent-establishment / local-labour-law view rather than just a contract-type field. _( src/lib/hr-classification.ts:45-61 )_
- **Inline per-row reclassification with reviewer stamp** - Each employee row is an independent POST form; saving writes the three values and stamps `classification_reviewed_at = NOW()` and `classification_reviewed_by = <admin user id>`. The table renders the last review date formatted in Asia/Kolkata, or the literal string `never` when `classification_reviewed_at` is null — so an unreviewed worker is visible at a glance. _( src/pages/admin/hr/classification/index.astro:15-25, 83-97 )_
- **Headcount roll-ups by classification and by country** - Computed in-page in JS over the fetched rows (not in SQL): `byClass` and `byCountry` counters, defaulting a null classification to `permanent` and a null country to `IN`. Classification counts are printed with the type's static risk word colour-coded (red = high, amber = medium, green = low). _( src/pages/admin/hr/classification/index.astro:32-38, 51-77 )_
- **Shared taxonomy reused at requisition time** - `src/pages/admin/hr/requisitions/index.astro:4` imports `{ CLASSIFICATIONS, COUNTRY_CODES }` from this module. In practice only `COUNTRY_CODES` is used, to populate the Country select on the "Raise new requisition" form (line 94); `CLASSIFICATIONS` is imported but never referenced — the requisition form uses its own separate `ENGAGEMENT_TYPES` constant for the Engagement select (line 85). So the intended link between "what we are hiring for" and "how the person will be classified" is not actually wired. _( src/pages/admin/hr/requisitions/index.astro:4, 85, 94 )_

**Tables**
- `hr_employees — the only table. No new table is created by this module; it adds five columns and one index to the existing employee table via `ensureClassificationSchema()`: `classification VARCHAR(40) DEFAULT 'permanent'`, `country_of_work VARCHAR(80) DEFAULT 'IN'`, `engaged_via VARCHAR(40)`, `classification_reviewed_at TIMESTAMPTZ`, `classification_reviewed_by UUID`, plus `hre_classification_idx ON hr_employees(classification, country_of_work)`.`
- `hr_employees (pre-existing columns this module reads or collides with): `id`, `full_name`, `email`, `is_active`, `updated_at`, and the separate legacy `employment_type VARCHAR(50)` that overlaps conceptually with `classification` but is never reconciled with it.`
- `Note on `classification_reviewed_by UUID`: declared with no `REFERENCES users(id)` foreign key, unlike other HR tables in this codebase (e.g. `hr_probation.confirmed_by_user_id UUID REFERENCES users(id)` in src/lib/hr-lifecycle.ts). Reviewer identity is unenforced.`

**Admin surface.** `/admin/hr/classification` (file: `src/pages/admin/hr/classification/index.astro`) — the only surface. Reached from the admin sidebar under "HR Management" as nav id `hr-classification`, label "Classification register" (`src/layouts/AdminLayout.astro:89`). Gate is minimal: `if (!user) redirect('/admin/login')`, `if (user.role === 'applicant') redirect('/portal')` — every other logged-in role can both view and edit. The page shows two roll-up cards (headcount "By classification" with a colour-coded risk word, headcount "By country") and then a table of up to 300 active employees (`WHERE is_active = true ORDER BY full_name LIMIT 300`). Each row is its own `<form method="POST">` with three selects — classification, country, engaged via — and a Save button. POST runs one statement: `UPDATE hr_employees SET classification=…, country_of_work=…, engaged_via=…, classification_reviewed_at=NOW(), classification_reviewed_by=${user.id}, updated_at=NOW() WHERE id=${id}`. There is no API route for classification anywhere under `src/pages/api/`.

**Employee surface.** none. No page under `src/pages/portal/` references `classification`, `country_of_work` or `engaged_via`, and the employee detail page `src/pages/admin/hr/employees/[id].astro` does not show or edit these fields either — it edits a separate, older free-text `employment_type` field instead. The worker is never shown, asked to confirm, or notified of their own classification.

**Not built / half-built**
- No criteria and no scoring. Despite the module framing misclassification as "the single biggest compliance risk", there is no test, questionnaire or score anywhere — no control test, no ABC test, no economic-reality factors, no per-person risk computation. `risk` is a static string attached to the *category* in `CLASSIFICATIONS`, so every contractor reads 'high risk' and every permanent employee reads 'low risk' regardless of the facts of the engagement. Nothing detects a contractor who is in practice an employee.
- No evidence capture. There is no place to attach the consulting agreement, contractor invoices, the EOR contract, or a rationale/note for why a person was classified the way they were — no `classification_notes`/`justification` column and no document link. If a labour authority asks 'why is this person a contractor', the system has nothing to show.
- No audit history. `classification_reviewed_at` / `classification_reviewed_by` are overwritten in place on each save. There is no `hr_classification_history` (or similar) table, so a reclassification from `permanent` to `contractor` leaves no trace of the prior value, who changed it, or when.
- No review enforcement. The page copy says "Review at least once a year" and the table prints `never`, but nothing acts on it: no due-date computation, no filter/sort by stale review, no reminder job, no entry in `src/pages/api/cron/`. The annual review is honour-system.
- Weak authorization. The only check is `user.role === 'applicant'` (`index.astro:9`). Any other authenticated admin-side role can reclassify any worker and thereby change their legal/benefits status. There is no HR-specific role check and no rate limit or CSRF token on the POST.
- No server-side validation of submitted values. `classification`, `country` and `engaged_via` are read straight off the FormData and written to the DB; nothing validates them against `CLASSIFICATIONS` / `COUNTRY_CODES` / `ENGAGEMENT_VIAS`. A crafted POST can write an arbitrary string (up to the VARCHAR(40)/(80) limits) — the UI then falls back to `CLASSIFICATIONS.permanent` when rendering an unknown value, silently mislabelling the row.
- Two overlapping concepts, unreconciled. `hr_employees` already carries a free-text `employment_type VARCHAR(50)` edited on the employee page with values `Full-Time | Part-Time | Contract | Internship | Freelance | Consultant | Apprenticeship | Fellowship` (`employees/[id].astro:501`), set at hire time by `transferApplicantToEmployee()` (`src/lib/hire-transfer.ts:34`). Nothing syncs it with the new `classification` column, nothing flags a contradiction (e.g. `employment_type='Full-Time'` + `classification='contractor'`), and the two are edited on different screens.
- No downstream consequence. `classification` is read by nothing else in the codebase — payroll, leave, wallet, onboarding and separation modules do not branch on it. Marking someone `eor` or `contractor` does not change what payroll or statutory logic does to them; it is a register, not a control.
- Hard scale ceiling and no search. The table is `LIMIT 300` with no pagination, no filter, no search box, and no way to view inactive employees (`WHERE is_active = true`). The roll-up counts are therefore counts of the first 300 rows, not of the whole workforce.
- No export and no API. No CSV/report output for counsel or an auditor, and no route under `src/pages/api/` — the data is only reachable through this one HTML page.
- No bulk action. Reclassifying a cohort (e.g. moving 20 people onto an EOR partner) is 20 individual form submissions.

### 6. Hiring Requisitions (HR / Workforce planning)

**Purpose.** A two-stage headcount-approval workflow where a manager raises a request to open a role (title, engagement type, headcount, budget band, KRAs, justification) and Finance then Leadership approve or reject it, intended as the gate that must pass before a role is opened for applications.

**Status:** partial

**Features that exist**

**Admin surface.** Exactly one route: `/admin/hr/requisitions` (src/pages/admin/hr/requisitions/index.astro), rendered in `AdminLayout` with `title=\"Hiring Requisitions\"`. It is a single self-POSTing page — no API endpoints, no detail route, no edit route (a repo-wide case-insensitive grep for \"requisition\" hits only this page, src/lib/hr-requisition.ts, the AdminLayout nav entry, docs/hrms-audit-scenario.md and src/data/aes-modules.json). On this page HR can: raise a requisition; filter the register by status; expand any of the latest 200 requisitions to read justification/KRAs/skills and the full approval trail; and, where status is pending_finance or pending_leadership, record an Approve or Reject decision with notes. Access control is thin: the page itself only enforces `if (!user) return Astro.redirect('/admin/login')` and `if (user.role === 'applicant') return Astro.redirect('/portal')`. The real restriction is the middleware section gate, where the prefix `['/admin/hr', 'hr']` (src/middleware.ts:25) maps to the 'hr' section key — `hr` is in ROLE_SECTIONS, but `recruiter`, `reviewer`, `department_head` and `marketing` are not, so those roles are redirected. Nothing distinguishes a Finance approver from a Leadership approver.

**Employee surface.** None. There is no employee/manager self-service view — no page under src/pages/portal touches requisitions, and no API route exposes them. A hiring manager who is not an admin user with the 'hr' section cannot raise or track a requisition; someone with admin access must raise it on their behalf, and the manager gets no notification of the outcome (there is no email, notification or webhook call anywhere in this module).

**Not built / half-built**
- THE LINK TO ROLES/OPENINGS IS NOT IMPLEMENTED. This is the central gap. `opened_role_id UUID` is declared in the CREATE TABLE (hr-requisition.ts:45, commented '-- references roles(id) once role is created from this req') but is never written, never read, and never selected by any code in the repo. There is no 'Create role from requisition' action, no function that flips `roles.is_open`, and no foreign key. The header comment 'Only after final approval can the role be marked is_open=true on roles' and the page copy 'Only approved requisitions become open roles' describe an intent the code does not enforce — `roles.isOpen` defaults to `true` (schema.ts:107) and /admin/roles can open a role with no requisition at all. An approved requisition today is a document, not a trigger.
- NO AUTHORIZATION ON WHO APPROVES. `decideRequisition()` takes `userId` but never checks a role; the page gate is only 'signed in and not an applicant'. Any user who can load the page can click Approve at the Finance stage and then Approve again at the Leadership stage — including the person who raised the requisition. There is no self-approval block and no separation of duties. (docs/hrms-audit-scenario.md independently flags hr-requisition.ts among the 'HR service modules [that] carry no authorization'.)
- NO STATUS GUARD ON THE UPDATE. Every UPDATE in decideRequisition() is `WHERE id=${opts.id}` with no `AND status=...` predicate, and `level` arrives as a client-controlled hidden form field. A replayed or hand-crafted POST can stamp finance approval onto an already-approved or already-rejected requisition, or reject an approved one. Combined with the repo-wide absence of CSRF tokens, the only protection is the session cookie's sameSite='lax'.
- NO THIRD (CPO) APPROVAL STAGE, despite the module comment promising 'Finance + Leadership + (CPO once we have one)'. The chain is hard-coded to two levels via a single `if (opts.level === 'finance') ... else ...`.
- THE 'withdrawn' STATUS IS DEAD. It exists in the column comment and in STATUS_TONE (index.astro:62) but no code path sets it, and it is not offered in the filter dropdown. A requester cannot withdraw their own requisition.
- NO EDIT, NO DELETE, NO REQUEST-CHANGES. Once submitted, a requisition is immutable except through the approve/reject stamps. A reviewer who wants a smaller budget band or lower headcount can only reject outright. `updated_at` is maintained but no history/versioning table exists.
- NO NOTIFICATIONS. Nothing emails or notifies Finance that a requisition is waiting, or tells the requester the outcome. Reviewers must visit the page and filter by status.
- NO PARTIAL/INCREMENTAL FULFILMENT. `headcount` is stored but nothing tracks how many of the N seats have been filled, so an approved requisition never closes itself.
- NO SERVER-SIDE VALIDATION BEYOND LENGTH SLICING. budget_min/max are not checked for min <= max; `parseFloat(...) || undefined` silently drops a legitimate 0; `level` is free text on the requisition while `roles.level` is a Postgres enum (levelEnum), and `engagement_type` is a plain VARCHAR(40) here versus the `engagementEnum` on roles — so requisition values cannot be copied to a role without mapping.
- NOT IN THE DRIZZLE SCHEMA / NO MIGRATION. The table is created lazily at request time inside a swallow-all `try/catch (_) {}`; if the DDL fails the module stays silently broken. src/lib/db/schema.ts contains no hiring_requisitions definition.
- MINOR UI DEFECTS: `CLASSIFICATIONS` is imported at index.astro:4 and never used; the layout is called with `activeNav=""` so the Requisitions sidebar item never highlights; the register is capped at LIMIT 200 with no pagination; `listRequisitions` does `SELECT *`.

### 7. Separations & Offboarding

**Purpose.** Records an employee's exit end-to-end — how it was initiated (resignation vs. termination), notice period, knowledge-transfer/asset/access clearance, exit interview, full-and-final settlement, and generation of relieving / experience / NOC letters.

**Status:** partial

**Features that exist**
- **Separation record (8 exit kinds)** - Opens a structured separation row against an active employee. `SEP_KINDS` defines exactly eight kinds: `resignation` (Employee-initiated), `performance` (After documented PIP failure), `misconduct` (After domestic inquiry), `retrenchment` (Role redundancy, not the person), `end_of_contract`, `medical`, `death` (Death in service — survivor benefits triggered), `other`. Also captures `initiated_by` = employee | employer, `notice_days` (form default 30; `openSeparation` falls back to 30), `last_working_day`, and free-text `reason` labelled 'will appear on relieving letter where applicable'. _( src/lib/hr-separation.ts (openSeparation, SEP_KINDS) -> src/pages/admin/hr/separations/index.astro (POST action=open) )_
- **5-stage separation status flow** - `status` column moves through `notice` -> `knowledge_transfer` -> `exit_interview` -> `settlement` -> `closed`, set manually from a <select> in each row. Each stage gets its own colour chip in the list. The only automatic transition is in `recordExitInterview`, which bumps status to `settlement` only if it was `notice` or `knowledge_transfer`. _( src/lib/hr-separation.ts:29-30, :114; src/pages/admin/hr/separations/index.astro:99 )_
- **Clearance checklist (4 booleans)** - Per-separation checkboxes for `kt_complete`, `assets_returned`, `access_revoked`, `alumni_invited`, plus a `eligibility_for_rehire` select (yes | no | with_caveats) and an `ff_amount` number field. All saved by one 'Save' form. _( src/pages/admin/hr/separations/index.astro:93-103 -> updateSeparation() )_
- **Structured exit interview (9 fixed questions)** - `EXIT_INTERVIEW_QUESTIONS` is a hardcoded 9-question set: why_leaving, what_worked, what_didnt, manager_feedback, team_feedback, policy_feedback, recommend, alumni_engagement, rehire_interest. Answers are inserted one row per question into `hr_exit_interview_responses` (truncated to 5000 chars), and `exit_interview_at` is stamped. The form is hidden once `exit_interview_at` is set — it can only be filled in once. _( src/lib/hr-separation.ts:75-85, recordExitInterview(); src/pages/admin/hr/separations/index.astro:105-117 (POST action=exit) )_
- **Separations list** - `listSeparations()` returns up to 200 separations joined to `hr_employees` for `full_name` / `email`, newest first, each as an expandable <details> row showing kind, last working day, status chip and initiation date. _( src/lib/hr-separation.ts:117-125; src/pages/admin/hr/separations/index.astro:77-123 )_
- **Second, separate offboarding flow on the employee record (Exit tab)** - An independent implementation stored on `hr_employees` itself, not on `hr_separations`. Four POST actions: `initiate_exit` (sets exit_type of resignation | termination | layoff, notice_period_days, last_working_day, exit_reason, stamps notice_served_at, sets employment_status='notice_period'); `update_exit_checklist` (7 booleans written to `offboarding_state` JSONB: exit_interview, knowledge_handover, assets_returned, access_revoked, dues_cleared, final_payroll, relieving_letter_issued, plus a `relieving_letter_url` Drive link); `mark_relieved`; `cancel_exit` (nulls every exit column and resets to active). _( src/pages/admin/hr/employees/[id].astro:116-187, Exit tab = panel-7 at :972 )_
- **Real access revocation on relieve** - This is the only place access is actually cut, not just ticked. `mark_relieved` sets hr_employees.is_active=false, employment_status='exited', exit_date (defaults to COALESCE(last_working_day, CURRENT_DATE)), then sets `users.is_active = false` for the linked user_id and deactivates `user_face_enrollments` rows so a rehire must re-enrol their face. _( src/pages/admin/hr/employees/[id].astro:153-176 )_
- **Rehire / reactivate** - A `reactivate` action restores is_active=true, employment_status='active' and nulls exit_type, notice_period_days, last_working_day, notice_served_at, exit_date, exit_reason, offboarding_state, relieving_letter_url. _( src/pages/admin/hr/employees/[id].astro:282-288 )_
- **Offboarding letter generator (relieving / experience / NOC)** - A live-preview document builder with three templates chosen by `f_docType`: Relieving Letter, Experience Letter, No-Objection Certificate. Form fields drive an on-page A4 preview on EduRankAI letterhead (ref number auto-seeded as 'OFF-2026-' + 6 random chars, signatory defaults to the logged-in user's name / 'Founder & CEO'). Relieving body asserts 'All financial and non-financial dues from both sides have been settled'; NOC renders a bulleted list from one-per-line input. Export via html2canvas -> jsPDF as 'EduRankAI-Relieving|Experience|NOC-<Name>.pdf', or browser Print (print CSS hides everything but #offDoc). _( src/pages/admin/hr/offboarding.astro; PDF libs from public/era/era-pdf.js )_
- **Automatic offboard on application status reversal** - A third path: if a job application goes from `hired` to `withdrawn` or `rejected`, `syncOffboardEmployee` sets the matching hr_employees row is_active=false, exit_date=today and exit_reason to the literal string 'resignation' (for withdrawn) or 'termination' (for rejected). _( src/lib/hr/sync.ts:14-22, :102-118 )_
- **Idempotent self-bootstrapping schema** - Neither surface needs a migration. `ensureSeparationSchema()` CREATEs `hr_separations`, indexes `sep_emp_idx` / `sep_status_idx` and `hr_exit_interview_responses` inside a swallowed try/catch, memoised in a module-level `ready` promise. The employee page separately runs `ALTER TABLE hr_employees ADD COLUMN IF NOT EXISTS exit_type, last_working_day, notice_period_days, notice_served_at, offboarding_state, relieving_letter_url` on every page load. _( src/lib/hr-separation.ts:10-62; src/pages/admin/hr/employees/[id].astro:13-23 )_

**Tables**
- `hr_separations — id, employee_id (FK hr_employees ON DELETE CASCADE), kind, initiated_by, initiated_at, notice_days, last_working_day, reason, pip_id, related_flag_id, status (default 'notice'), kt_complete, assets_returned, access_revoked, exit_interview_at, exit_interview_notes, ff_amount DECIMAL(14,2), ff_currency (default 'INR'), ff_paid_at, relieving_letter_url, experience_letter_url, alumni_invited, alumni_accepted, eligibility_for_rehire, created_at, updated_at`

**Admin surface.** Three admin routes, all gated only by `if (!user) redirect('/admin/login')` and `if (user.role === 'applicant') redirect('/portal')` — any non-applicant admin role can open a separation, record an exit interview or relieve an employee. **Employee surface.** None. There is no employee-facing or manager-facing surface anywhere in the module: no resignation submission form, no notice-period countdown in /portal, no clearance task list for the departing employee, no self-service exit-interview form, and no place for the employee to download their own relieving or experience letter. Every field — including all nine exit-interview answers — is typed by an HR admin. (The only portal mention of a relieving letter, src/pages/portal/employee/onboarding.astro:99, is an upload slot for the PREVIOUS employer's NOC/relieving letter during joining, not an exit artefact.)

**Not built / half-built**
- THREE parallel, non-communicating offboarding stores. Opening a separation in /admin/hr/separations does not touch hr_employees at all — it never sets employment_status='notice_period', never sets is_active=false, and never deactivates the login. Conversely the Exit tab on /admin/hr/employees/[id] never creates an hr_separations row. A third path, syncOffboardEmployee (src/lib/hr/sync.ts:102), writes exit_date/exit_reason directly. The same employee can simultaneously be 'closed' in hr_separations and still fully active in hr_employees, or vice versa.
- The clearance checkboxes on /admin/hr/separations cannot be UN-ticked. index.astro:32 does `const v = fd.get(k); if (v !== null) patch[k] = ...` — an unchecked HTML checkbox is simply absent from the FormData, so `v` is null and the key is skipped. kt_complete, assets_returned, access_revoked and alumni_invited are therefore one-way latches. (The equivalent checklist on the Exit tab, which uses `fd.get(k) === 'on'`, does not have this bug.)
- The `access_revoked` checkbox in the separations flow revokes nothing — it is a self-attested boolean. Real revocation (users.is_active=false, user_face_enrollments deactivated) exists only in `mark_relieved` on the employee Exit tab.
- Exit interview answers are write-only. `hr_exit_interview_responses` is INSERTed by recordExitInterview() and never SELECTed anywhere in the codebase. After submission the UI shows only a date stamp ('Exit interview · 12 Mar'); the nine answers cannot be read back, reported on, exported or aggregated in any surface. The `exit_interview_notes` column is in the updateSeparation allow-list but no form ever posts it.
- Notice period is a stored number only — no computation. `notice_days` / `notice_period_days` are never used to derive or validate `last_working_day`, there is no days-remaining countdown, no notice-shortfall or buyout calculation, and no reminder/alert as the last working day approaches.
- Full & final settlement is a single unvalidated number. `ff_amount` is typed by hand with no breakdown (no gratuity, leave encashment, notice recovery, advances or deductions), no link to the payroll module, and no approval step. `ff_currency` and `ff_paid_at` exist as columns and are in the updateSeparation allow-list but no form field ever sets them — F&F can never be marked paid through the UI.
- The letter generator at /admin/hr/offboarding persists nothing. It reads no employee from the database (name, ID, role, department, join date and last working day are all retyped by hand), and it writes nothing back — the generated PDF is not stored, and `hr_separations.relieving_letter_url` / `experience_letter_url` are not populated by it. The relieving template hardcodes the claim 'All financial and non-financial dues from both sides have been settled' regardless of whether any clearance item is actually ticked.
- No document is uploaded or hosted. `relieving_letter_url` and `experience_letter_url` are free-text URL fields expecting a manually pasted external link (the Exit-tab field is literally placeholdered 'https://drive.google.com/...'), even though the project uses @vercel/blob elsewhere. `experience_letter_url` has no form input at all on the separations page despite being in the allow-list.
- The advertised PIP and misconduct chaining is not wired. `pip_id` and `related_flag_id` are declared on hr_separations, but openSeparation is called from exactly one place (separations/index.astro:20) and that call passes neither — both columns are permanently NULL. There is no button on a failed PIP or a misconduct flag that opens a separation, and no FK constraint to hr_pips / hr_employee_flags.
- The alumni programme is a checkbox, nothing more. `alumni_invited` is a manual tick with no invitation being sent, and `alumni_accepted` has no UI field at all. No alumni table, page or route exists.
- `listSeparations(filterStatus?)` supports status filtering but the page always calls `listSeparations()` with no argument — there is no filter, search, sort or pagination in the UI, and the list is hard-capped at LIMIT 200.
- No API endpoints, no audit trail and no notifications. There is no src/pages/api/**/separation* route; every action is a form POST rendered server-side. Nothing writes to an audit log when a separation is opened, a status is changed or an employee is relieved, and no email is sent to the employee, their manager, IT or finance at any stage.
- Kind vocabularies disagree across the two flows: hr_separations.kind has 8 values (resignation | performance | misconduct | retrenchment | end_of_contract | medical | death | other) while hr_employees.exit_type offers 3 (resignation | termination | layoff) and syncOffboardEmployee writes free strings 'resignation' / 'termination'. Neither column is constrained by a CHECK or enum.
- ensureSeparationSchema() wraps every DDL statement in `catch (_) {}` and memoises the promise, so a failed table creation is silent and never retried for the life of the process — the page then renders an empty list rather than an error.
- No death-in-service handling beyond the label. `death` is an offered kind whose description reads 'Survivor benefits triggered', but no nominee, survivor-benefit or insurance-claim field or flow exists.
- Minor: in offboarding.astro:280, if the PDF library has not loaded the handler alerts and returns after already setting `btn.disabled = true` and the label to 'Generating PDF...', leaving the Download button permanently stuck.

### 9. Conduct Flags & HR Support

**Purpose.** Two unrelated capabilities that share only the "HR" prefix: `hr-flags.ts` is an internal, append-only disciplinary record for employees (three severity levels with an automatic three-strike escalation), while `hr-support.ts` is an outward-facing, paid 25 CHF service where a job candidate books a video call with an HR rep and converses with them in a threaded message stream — it is NOT an internal helpdesk or IT ticketing system.

**Status:** partial

**Features that exist**
- **Breach catalogue (BREACH_TYPES)** - A frozen 10-entry catalogue keyed by breach slug, each carrying a `level` (1-3), a `label`, and a default `action` string. Level 1: `late_deliverable`, `hours_breach`, `communication_failure`. Level 2: `misrepresentation`, `plagiarism_ai_abuse`, `unprofessional_conduct`. Level 3: `confidentiality_breach`, `ip_theft`, `undisclosed_coi`, `harassment`. The source comment states these are "exact strings from policy v2.0" (Recruitment & Work Policy v2.0). The action string is the policy consequence, e.g. `harassment` = 'Immediate termination · Formal report · Possible legal referral'. _( src/lib/hr-flags.ts:37-51 )_
- **Raise a flag (raiseFlag)** - Inserts one row into `hr_employee_flags` with the level looked up from the catalogue — the caller cannot choose the severity, it is derived from the breach type. `action_taken` falls back to the policy's default action string when the operator leaves the field blank. Records `flagged_by_user_id` and `flagged_by_name`. Returns `{ ok, flagId, autoEscalated }`. _( src/lib/hr-flags.ts:54-95; invoked from src/pages/admin/hr/employees/[id].astro:256-279 (form action `raise_flag`) )_
- **Three-strike auto-escalation** - After a Level 1 flag is written, it counts that employee's Level 1 rows; when the count is >= 3 and divisible by 3, it writes a SECOND row at level 2 with `breach_type = 'three_strike_escalation'`, `is_escalation = true`, `escalated_from_count` = the count, and `flagged_by_name = 'System (auto-escalation)'`. The description cites 'Recruitment & Work Policy §05 three-strike rule'. The admin page surfaces this as a 'Three-strike threshold reached' pill plus the success message 'Auto-escalated to Level 2 review (three-strike rule).' _( src/lib/hr-flags.ts:74-89; badge at src/pages/admin/hr/employees/[id].astro:826-828 )_
- **Per-employee flag history (listFlags)** - Returns all flags for one employee, newest first, including the appeal fields. Rendered as a colour-coded timeline (blue L1 / amber L2 / red L3) with description, action taken, who recorded it, and an appeal line when `appeal_status` is non-null. Rows whose breach_type is not in the catalogue (i.e. `three_strike_escalation`) fall back to displaying the raw slug. _( src/lib/hr-flags.ts:97-105; UI at src/pages/admin/hr/employees/[id].astro:317-318 and 819-875 (panel id `panel-flags`) )_
- **Flag level counters** - The panel header computes and shows `flags.length` total plus L1 (excluding auto-escalation rows), L2 and L3 counts for that employee. _( src/pages/admin/hr/employees/[id].astro:328-330, 824 )_
- **Self-bootstrapping flag schema** - `ensureFlagsSchema()` issues CREATE TABLE IF NOT EXISTS plus two indexes (`hr_flags_emp_idx`, `hr_flags_level_idx`) on first call, memoized in a module-level promise. All errors are silently swallowed (`catch (_) {}`), so a failed bootstrap is invisible and surfaces later as a query error. _( src/lib/hr-flags.ts:9-34 )_
- **Paid HR application-support booking** - A public candidate form collects name, email, phone, `support_kind`, a free-text `situation` (min 30 chars, max 5000), optional target role slug and timezone, plus a mandatory slot chosen from a server-rendered calendar of the next 14 days x 3 fixed IST slots (10:00 / 14:00 / 17:00). On submit it creates the row via `createHrSupportRequest()`, then a direct UPDATE in the page writes `requested_slot_at` / `requested_slot_label`, and redirects to the payment page. _( src/lib/hr-support.ts:83-97; page src/pages/careers/hr-support.astro (slot generator 44-64, POST handler 8-42) )_
- **Support-kind catalogue (SUPPORT_KINDS)** - Seven kinds with label + first-person description: `application_strategy`, `role_selection`, `interview_prep`, `profile_review`, `cv_review`, `portfolio_review`, `general`. _( src/lib/hr-support.ts:149-157 )_
- **Razorpay payment for the 25 CHF fee** - The payment page creates a Razorpay order (`receipt: 'hrs_' + id`, notes tagged `kind: 'hr_application_support'`) and stores `razorpay_order_id`. The amount is HARDCODED at `2500 * 100` paise (INR 2,500) for the advertised CHF 25, with an inline comment 'assume ~93 INR/CHF spot ... Adjust later via the same FX helper Tests uses'. Checkout runs in-page; the handler POSTs to the verify endpoint. _( src/pages/payments/hr-support/[id].astro:20-38, 84-112 )_
- **Payment signature verification** - Recomputes HMAC-SHA256 of `order_id + '|' + payment_id` with `RAZORPAY_KEY_SECRET`; on match sets `payment_status = 'paid'`, `razorpay_payment_id`, `paid_at`. _( src/pages/api/payments/hr-support/verify.ts:8-34 )_
- **Two-sided message thread** - `hr_support_messages` holds admin replies and candidate responses in one chronological stream (`sender_role` = admin | candidate | system), with an optional `meta` JSONB carrying the template key and meet link. Admin works it at /admin/hr-support/[id]; the candidate sees the same thread and can reply at /portal/hr-support/[id]. Both views cap at 500 messages. _( src/lib/hr-support.ts:53-65 (schema); admin src/pages/admin/hr-support/[id].astro:23-42, 64-67, 107-122; candidate src/pages/portal/hr-support/[id].astro:17-30, 39-42 )_
- **Quick-reply templates (REPLY_TEMPLATES)** - Six pre-written replies, each with `label`, colour `tone`, a `body` containing {{name}}/{{slot}}/{{meet_link}}/{{id}} placeholders, and an optional `statusUpdate`: `confirm_meet` (-> scheduled), `reschedule_request` (-> reschedule_pending), `cancelled_refund` (-> cancelled), `no_show` (-> no_show), `completed` (-> completed), `general_reply` (blank body, no status). `renderTemplate()` fills the placeholders; the admin page pre-renders all six server-side so one click drops the finished text into the textarea. _( src/lib/hr-support.ts:103-147; pre-render at src/pages/admin/hr-support/[id].astro:75-81, buttons 132-161 )_
- **Status side-effects on reply** - Sending a reply with a `status_update` applies: `completed` sets `completed_at = NOW()`; `cancelled` sets `payment_status = 'cancelled'`; `refunded` sets `payment_status = 'refunded'`; `no_show` and `reschedule_pending` merely append a bracketed tag to the free-text `notes` column. Supplying a meet link sets `meet_link`, `meet_scheduled_at` (via COALESCE, so never overwritten) and `hr_rep_user_id`. _( src/pages/admin/hr-support/[id].astro:34-40 )_
- **Admin request queue** - Lists up to 300 requests newest-first with five KPI tiles (total, awaiting payment, paid, scheduled, completed), free-text search over candidate_name/candidate_email (ILIKE), a payment filter (pending/paid/refunded/cancelled) and a derived-status filter (open/scheduled/completed). _( src/pages/admin/hr-support/index.astro:14-97 )_
- **Admin request controls** - Three further POST actions on the detail page: `mark_paid` (manual override setting payment_status='paid'), `modify_slot` (datetime-local + label, overwrites requested_slot_at/label), and `edit_notes` (internal notes, max 10000 chars, explicitly labelled 'Not visible to candidate'). A right-hand facts panel exposes the Razorpay order and payment IDs. _( src/pages/admin/hr-support/[id].astro:43-57, 166-205 )_
- **Candidate self-service portal** - /portal/hr-support lists the signed-in user's own sessions, scoped by `candidate_email = user.email OR user_id = user.id`, each with a per-request message-count subquery and a 'Meet link ready' hint. The detail page renders the thread, a prominent Join-the-call button when `meet_link` is set, and a reply box. _( src/pages/portal/hr-support/index.astro:13-20; src/pages/portal/hr-support/[id].astro:32-37, 65-70 )_
- **Derived status model** - There is no status column. Every surface recomputes the same expression: completed_at ? 'completed' : meet_scheduled_at ? 'scheduled' : payment_status==='paid' ? 'paid' : 'awaiting payment'. The admin list re-encodes the same logic as a SQL CASE for filtering. _( src/pages/admin/hr-support/index.astro:27, 80; src/pages/admin/hr-support/[id].astro:71; src/pages/portal/hr-support/index.astro:46; src/pages/portal/hr-support/[id].astro:46 )_
- **Self-bootstrapping support schema** - `ensureHrSupportSchema()` creates both tables, then runs two ALTER TABLE ... ADD COLUMN IF NOT EXISTS for `requested_slot_at` / `requested_slot_label` (added after the original table shipped), plus three indexes: `hras_status_idx`, `hras_email_idx`, and a partial `hras_slot_idx` on requested_slot_at WHERE NOT NULL. Errors swallowed the same way as the flags module. _( src/lib/hr-support.ts:9-69 )_

### 10. Employee Self-Service (portal) — `/portal/employee` + `/portal/employee/wallet` + `/portal/employee/leave` (+ `/portal/employee/onboarding`)

**Purpose.** A mobile-first "My HR" surface where a person already recorded in `hr_employees` clocks in/out with a selfie and GPS, logs their working hours and daily report, applies for leave, manages their salary wallet/bank account, and views their payslip history — all scoped to their own record only.

**Status:** partial

**Features that exist**
- **Auto-link user account to employee record** - On every page load it looks up `hr_employees WHERE user_id = <session user id> AND is_active = true`. If that misses, it falls back to `work_email OR personal_email OR email = <session email>` and then permanently writes `UPDATE hr_employees SET user_id = ...`. - No holiday calendar, no team/leave calendar, no manager view of who is out, and leave allowances are hard-coded constants in `LEAVE_TYPES` (src/lib/hr-leave.ts:11-16) rather than per-employee or per-policy data — no pro-rating for joiners, no carry-forward, no weekend/holiday exclusion (`daysBetween` counts raw inclusive calendar days).
- No expense claims or reimbursement submission. `hr_wallet_txn.kind` includes `'reimbursement'` and the wallet page says "Salary and reimbursements appear here", but there is no employee-facing form to claim one — only HR can credit.
- No documents surface: no offer letter, contract, Form 16 / tax documents, experience letter or ID card in the portal.
- No performance, goals, appraisal, training or e-learning self-service inside My HR (`/admin/hr/performance.astro` and `/admin/hr/training.astro` exist, admin-only, with no employee counterpart).
- No employee HR helpdesk. `/portal/hr-support` exists but is the paid *job-application* support product for candidates, not an internal ticket desk, and it is not linked from `/portal/employee`.
- No CSRF token on any of the POST forms in this module, and no rate limiting on clock events.
- Error handling is silent and misleading: the employee lookup and every read is wrapped in bare `catch(e){}` (employee.astro:20, 32, 143, 152, 163, 178, 205), so a DB failure renders as "No employee profile found" or as empty cards. Server errors are surfaced through the same `?msg=` banner that is always styled green as a success (:245), so "Could not save report: ..." appears as a success message.
- DDL runs on the request path: `ALTER TABLE hr_clock_events ADD COLUMN IF NOT EXISTS face_photo TEXT` fires on every clock POST (:50) and `CREATE TABLE IF NOT EXISTS hr_task_log` on every clock-out (:76).
- Face photos are stored as base64 data URIs inline in a TEXT column with lat/lon/IP/device fingerprint per event, with no retention policy, no restriction on who can read them and no access audit (also flagged in docs/hrms-audit-scenario.md:476).
- `approverRole()` builds its reporting-manager check with `sql.raw` and naive quote-stripping string interpolation (src/lib/hr-wallet.ts:115) instead of a parameterised query — it governs who may approve this module's leave and withdrawal requests.
- Time-log entries can overlap each other and can exceed 24h/day; there is no validation against the clock-in window and no edit (only add and delete).
- There is a separate offline `/portal/worklog` page (IndexedDB-backed, `data-offline="worklog"`) that duplicates the time-log idea, but it is not linked from `/portal/employee` and does not write to `hr_time_logs`.
- `/portal/employee` itself has no offline support: every clock action is a full-page form POST, so an employee with no signal in the field cannot clock in at all.

- The export ignores every filter. `/api/export/employees` re-queries the whole table with `LIMIT 5000` and never reads `q`, `status` or `page`. What HR sees on screen and what lands in the CSV are unrelated, and a 5001st employee is silently dropped with no truncation warning.
- The export bypasses section RBAC. The middleware permission gate only runs for `path.startsWith('/admin/')` (src/middleware.ts:265); `/api/export/employees` checks only `!user || user.role === 'applicant'` (src/pages/api/export/employees.ts:16-18). A staff user whose role denies the `employees` section can still fetch the CSV — including the `Base Salary` and `Currency` columns.
- Two competing definitions of "active". The dashboard counts `is_active = true`; the directory list and its stat tiles filter on `employment_status`. `src/lib/hr/sync.ts:59-77` inserts hired applicants with `is_active = true` but never sets `employment_status`, so those rows count toward the dashboard's Active Employees tile yet are invisible under the directory's default `status=active` filter (NULL never equals 'active').
- Employee-code generation is not collision-safe. Both `src/pages/admin/hr/employees/index.astro:54-57` and `src/lib/hr/sync.ts:48-51` derive the code from `COUNT(*) + 1`. Any deactivation-then-add, or two concurrent adds, reissues an existing `ERA-EMP-nnnn`, and there is no unique constraint or retry.
- No duplicate-email or duplicate-person check on add. `email` is only trimmed and lower-cased; the same person can be added repeatedly.
- The add form cannot set fields the table has. `department_id` and `personal_email` are columns the module reads or that other code writes, but the modal offers no department picker; there is likewise no manager/reporting-to field, no location, and no user-account creation, so a new employee has no `user_id` link and cannot see `/portal/employee` until the email-matching fallback fires.
- `hr_employees` has no Drizzle schema entry. src/lib/db/schema.ts contains no `hr_employees`/`hrEmployees`. The pgTable defined at src/pages/admin/hr/employees/index.astro:12-29 is dead code — it is never referenced (all queries are raw `db.execute(sql...)`), and the imports `eq, ilike, or, desc` on line 4 are unused. It is also documented as unmigrated in docs/hrms-audit-scenario.md (gap G13).
- The "New (30d)" tile is misnamed and over-counts. Its SQL alias is `new_this_month` but the predicate is `joining_date >= NOW() - INTERVAL '30 days'` with no upper bound, so employees with a future joining date are included.
- Error handling leaks and hides. The directory surfaces raw driver text to the browser via `errorMsg = 'Error: ' + (err.message || 'Unknown')`, the export returns `'Export failed: ' + e.message` with a 500, and the dashboard's `catch(e) {}` turns a database outage into a plausible-looking all-zero dashboard.
- No sort control, no column chooser, no saved views, and no filtering by department, employment type, work mode or joining-date range despite all four columns being stored and displayed.
- Pagination links are built by string interpolation without `encodeURIComponent` (lines 233 and 235), so a search term containing `&` or `#` breaks Prev/Next.
- No audit trail. Activate/deactivate and add write only `updated_at`; nothing records which admin performed the change.
- `Asset Management` is advertised on the dashboard as a `planned` module pointing at `/admin/hr/assets`. No such page exists under src/pages/admin/hr/ — the card is deliberately inert (`href='#'`).
- The dashboard grid is out of sync with the product. Six HR surfaces that exist and are in the sidebar — `/admin/hr/onboarding`, `/admin/hr/requisitions`, `/admin/hr/bgv`, `/admin/hr/classification`, `/admin/hr/separations`, `/admin/hr/wallet` — have no card on `/admin/hr`, because the `modules` array is hand-maintained.

### 3.1 Leave Management

**Purpose.** Lets an employee apply for time off against a fixed annual per-type allowance, and lets an admin/HR user approve or reject it. Balances are recomputed from the request history on every page load; there is no stored balance ledger.

**Features that exist**
- Hardcoded leave catalogue in `LEAVE_TYPES` (`src/lib/hr-leave.ts:11-17`): `casual` (12), `sick` (12), `earned` (15), `unpaid` (0). No admin CRUD, no per-grade/location policy.
- `ensureLeaveSchema()` lazily creates `hr_leave_request` plus indexes `hr_leave_status` and `hr_leave_emp` on first use.
- `getBalances(employeeId, year?)` — one GROUP BY summing `days` for `approved` into `used` and `pending` into `pending`, filtered by `EXTRACT(YEAR FROM start_date)`; `remaining = max(0, allowance - used - pending)`; `unpaid` returns `Infinity`.
- `applyLeave()` — validates the type, counts inclusive calendar days via `daysBetween()`, rejects `end < start`, re-checks the balance and rejects with `Only N <type> day(s) remaining this year.`, then inserts a `pending` row.
- `cancelLeave(id, employeeId)` — employee withdraws their own still-pending request (scoped by both employee id and `status='pending'`).
- `decideLeave(id, user, decision, note)` — refuses non-pending rows (`Already <status>.`), re-authorises via `approverRole`, writes `status`, `decided_by`, `decided_by_role`, `decided_at`, `decision_note`.
- `approverRole(user, employeeId)` (imported from `src/lib/hr-wallet.ts:107-119`) returns `super_admin` / `admin` / `hr_head` (substring test on the role) / `reporting_manager`, else null. Shared with wallet withdrawals.
- Admin queue: pending-first list (LIMIT 120) with inline Approve/Reject and a free-text note, plus a read-only "Recent" list of 25 decided rows showing `decided_by_role`. POST-redirect-GET via `?msg=`.

**Tables owned**
- `hr_leave_request` (singular) — the only leave table with DDL in the repo. `id`, `employee_id`, `leave_type`, `start_date`, `end_date`, `days INT`, `reason`, `status` (default `pending`), `requested_at`, `decided_by`, `decided_by_role`, `decided_at`, `decision_note`. No FK to `hr_employees`, no status CHECK, no overlap constraint.

**Tables read but not owned (orphaned second schema, no DDL anywhere)**
- `hr_leave_requests` (plural, `leave_type_id`, `applied_at`), `hr_leave_types` (`name`, `code`, `is_paid`, `days_per_year`, `is_active`), `hr_leave_balances` (`allocated`, `used`, `carried_forward`, unique on `(employee_id, leave_type_id, year)`).
- `hr_employees` — joined for display and the approver check.

**Admin surface.** `/admin/hr/leave` (`src/pages/admin/hr/leave/index.astro`), sidebar id `hr-leave`, icon `calendar`. Gate: signed in and `role === 'super_admin' || role === 'admin' || role.indexOf('hr') >= 0`. One unscoped global queue — no filter, search, pagination, bulk action, allocation screen, leave-type editor or holiday calendar. **Employee surface.** `/portal/employee/leave` — balance cards for the three allowance-bearing types, an apply form (type / from / to / reason, with honest copy "Counts calendar days between the dates (inclusive)"), the last 60 requests with status pills, and a `×` cancel control on pending rows. Linked from the `/portal/employee` hub.

### 3.2 Attendance & Time Tracking

**Purpose.** Records who worked when — employee self-service clock in/out/break with GPS and an optional selfie, plus an HR day view where attendance can be marked, corrected, bulk-set and read alongside each employee's hourly time log and written daily report.

### 3.3 Payroll & Payslips

**Purpose.** Monthly India-centric payroll: HR sets a per-employee salary structure, generates a month/year run that auto-computes earnings, statutory deductions and net pay into per-employee payslip rows, moves the run draft → approved → paid, and renders a printable HTML payslip.

**Features that exist**
- Effective-dated salary structure editor: `currency`, `basic`, `hra`, `da`, `specialAllowance`, `transportAllowance`, `medicalAllowance`, `otherAllowances`, manual monthly `tds`. Saving closes the open row (`effective_to = CURRENT_DATE - INTERVAL '1 day'`), inserts a new one, and writes gross back to `hr_employees.base_salary`/`currency`.
- Run creation: one `hr_payroll_runs` row per month+year (duplicate rejected), then a loop over active employees LEFT JOINed to the effective salary structure, one `hr_payslips` INSERT each with `ON CONFLICT (payroll_run_id, employee_id) DO NOTHING`, then a totals UPDATE.
- Hardcoded statutory engine: PF employee = `min(basic * 0.12, 1800)`; ESIC = `gross * 0.0075` only when `gross < 21000`; professional tax = 200 above 15000, 150 above 10000, else 0. Formulas duplicated in the salary-save path.
- Fallback defaults when no structure exists: basic from `hr_employees.base_salary`, HRA = `basic * 0.4`, transport 1600, medical 1250.
- Attendance snapshot per payslip (`days_worked`, `days_leave`, `days_absent`) — displayed only.
- Approval workflow: `approve_run` (status, `approved_by`, `approved_at`, cascade to payslips) and `mark_paid` (status, `paid_at`, cascade) behind a JS confirm "This cannot be undone".
- Runs list (latest 24) with employee count and Gross/Deductions/Net tiles; Payslips tab with a run picker and per-employee cards.
- Printable payslip at `/admin/hr/payslip/[id]` — self-contained HTML (not PDF), header, status badge, employee/pay-period block, 3-box attendance grid, side-by-side Earnings/Deductions tables (rows only when > 0), Net Pay banner, "computer-generated payslip" footer, `window.onload → window.print()`, `@media print` rules.
- Employee payslip history: last 12 payslips with month/year and net amount plus a Download button.
- Dashboard counters: "Payroll Drafts" (`COUNT(*) WHERE status='draft'`), HR hub tile, `/admin` Run Payroll quick action, sidebar item, admin search entry.
- Currency labelling from `['INR','USD','GBP','EUR','SGD','AED']`.
- Access control: page and API both hard-restrict to `['super_admin','hr']`; middleware section key `payroll`.

**Tables owned** — `hr_payroll_runs`, `hr_payslips`, `hr_salary_structures` (full column lists in the inventory; **no DDL in the repo and no Drizzle schema entry** for any of the three).
**Tables read/written** — `hr_employees` (read/written), `hr_attendance` (read-only), `departments` (read-only LEFT JOIN for the payslip header).

**Admin surface.** `/admin/hr/payroll` with `?tab=runs|payslips|salary`, plus a `+ Run Payroll` modal (month, year, notes). Salary tab is capped at the first 100 active employees. `/admin/hr/payslip/[id]` is a GET `APIRoute` returning the printable document.

**Employee surface.** A read-only "Payslips" card on `/portal/employee` with a Download button per row — **which does not work for ordinary employees**, because it links into `/admin/hr/payslip/{id}` and that route 401s for anything other than `super_admin`/`hr` (and middleware redirects the user before that). No employee-facing salary-structure view, no tax declaration, no investment-proof upload, no payslip email.

**Completeness: partial.**

### 3.4 HR Wallet / Payouts

**Purpose.** A double-entry employee wallet: HR credits salary/bonus/reimbursement, the employee registers a bank account and requests a withdrawal, an approver (reporting manager, HR head, admin or super-admin) decides, and HR releases the money via RazorpayX Payouts or records a manual settlement.

**Features that exist**
- `ensureWalletSchema()` creates `hr_wallet_txn`, `hr_bank_account`, `hr_withdrawal` plus indexes `hr_wallet_txn_emp` and `hr_withdrawal_status` on first use.
- `getBalance()` returns `{ balance, currency, pending, available }`; balance is the signed sum of the ledger, pending is the sum of `pending`+`approved` withdrawals, `available = max(0, balance - pending)`.
- `credit(employeeId, amount, kind, note, createdBy, ref?)` — kinds offered in the UI: `salary`, `bonus`, `reimbursement`, `adjustment`. Silently no-ops if amount ≤ 0.
- `listTxns()` — last N entries, credits rendered green `+`, debits red `−`.
- `addBankAccount()` — demotes existing rows, inserts the new one as primary; inputs truncated (holder 120, account 30 whitespace-stripped, IFSC 15 uppercased, bank 120, UPI 80). `listBankAccounts()` adds a masked `••••1234` field.
- `requestWithdrawal()` — re-reads balance, rejects ≤ 0 or above `available` ("Amount exceeds your available balance (₹…)").
- `listWithdrawals({ employeeId?, status? })` — employee mode (LIMIT 50) and admin mode joining employee and bank details, pending-first (LIMIT 100).
- `approverRole()` — the shared chain also used by leave.
- `decideWithdrawal()` — refuses non-pending, stamps decision fields.
- `payWithdrawal()` — requires `approved` status and a role of `admin` / `super_admin` / `hr_head` (reporting managers explicitly cannot release money). With RazorpayX env vars set, POSTs to `https://api.razorpay.com/v1/payouts` (HTTP Basic, amount in paise, `mode: 'IMPS'`, inline `fund_account`, `queue_if_low_balance: true`, narration "Salary withdrawal"); otherwise takes a manual UTR. On success sets `status='paid'`, `payout_ref`, `paid_at` and inserts the offsetting `debit` / `withdrawal` ledger row.
- Payout-channel banner driven by `RAZORPAYX_ACCOUNT_NUMBER`, swapping the button between "Release via RazorpayX" and "Mark paid" + UTR field.
- Admin triage board bucketed into Pending / Approved-ready-to-pay / Recent (last 20 settled).
- Employee profile auto-link with `user_id` back-fill.

**Tables owned** — `hr_wallet_txn`, `hr_bank_account`, `hr_withdrawal` (full columns in the inventory). **Read/joined** — `hr_employees`.

**Admin surface.** `/admin/hr/wallet` in `AdminLayout`, title "Wallet & payouts". Gate: `super_admin`, `admin`, or a role containing `hr`. Three PRG POST actions: `disburse`, `decide`, `pay`. **Not linked from any navigation** — reachable only by typing the URL.

**Employee surface.** `/portal/employee/wallet` — dark balance card (available in large type, plus "In wallet" and "Pending withdrawal"), withdraw form capped at `available` via the `max` attribute, withdrawal history with colour-coded pills, bank accounts with masked numbers and an add/change form (holder + account number + IFSC or UPI), and the last 20 ledger entries. The employee never sees the approver's identity.

**Completeness: partial.**

### 3.6 Hiring Requisitions

**Purpose.** A two-stage headcount-approval workflow: a manager raises a request to open a role (title, engagement type, headcount, budget band, KRAs, justification), then Finance and then Leadership approve or reject it.

**Features that exist**
- Raise form (`<details> + Raise new requisition`) POSTing `action=create`: `role_title` (required, 200 chars), `department`, `level`, `engagement_type`, `country`, `work_mode`, `headcount`, `justification` (required, 4000 chars), `budget_band_min/max`, `budget_currency`, `target_join_by`, `kras`, `skills_required`. Requester identity taken server-side from `Astro.locals.user`. Success banner: "Requisition #<first 8 chars> raised. Routed to Finance for approval."
- Two-stage chain in `decideRequisition()`: approve at `finance` → `pending_leadership` (stamps `finance_approved_by/at/notes`); approve at `leadership` → `approved` (stamps the leadership trio).
- Reject at either stage → `status='rejected'`, `rejection_reason` (falls back to `'No reason given'`), `rejected_at`. Terminal — no re-open, resubmit or appeal.
- Contextual decision panel that only renders for `pending_finance`/`pending_leadership` and derives the hidden `level` from the current status, so the reviewer never picks a stage.
- `ENGAGEMENT_TYPES` — 6 classifications with descriptions, including an explicit contractor warning ("pay on invoice, NO control over hours/tools") and EOR guidance.
- Register: `SELECT *` ordered by `created_at DESC`, LIMIT 200, as expandable cards showing justification / KRAs / skills plus timestamped Finance notes, Leadership notes and rejection reason — an on-screen approval trail. GET status filter (All / Pending Finance / Pending Leadership / Approved / Rejected).
- `ensureRequisitionSchema()` creates the table and two indexes at first request, DDL errors swallowed.
- Sidebar entry `hr-requisitions`, label "Requisitions", icon `document`.

**Tables owned** — `hiring_requisitions` (full column list in the inventory; status enum `pending_finance | pending_leadership | approved | rejected | withdrawn`; `budget_currency` defaults to `'CHF'`), with indexes `hr_req_status_idx` and `hr_req_dept_idx`.
**Intended downstream target** — `roles` (`src/lib/db/schema.ts:94-119`). There is no `requisition_id` on `roles`, and nothing writes `hiring_requisitions.opened_role_id`.

**Admin surface.** Exactly one route: `/admin/hr/requisitions`. Single self-POSTing page — no API endpoints, no detail route, no edit route. Page-level gate is only "signed in and not `applicant`"; the real restriction is the middleware `hr` section key, which excludes `recruiter`, `reviewer`, `department_head` and `marketing`. Nothing distinguishes a Finance approver from a Leadership approver.

**Employee surface.** **None.** No portal page and no API route. A hiring manager without admin `hr` access cannot raise or track a requisition, and gets no notification of the outcome.

**Completeness: partial.**

## 4. What an employee can do for themselves

All of this lives at `/portal/employee`, `/portal/employee/leave` and `/portal/employee/wallet`.

**Time**
- Clock in, choosing a work mode (`remote` / `office` / `hybrid` / `field`).
- Take a break and resume work.
- Clock out, with a task wrap-up: what got done, status (`done` / `pending` / `blocked`), a 0-100% completion slider and free-text details.
- Have GPS coordinates and accuracy captured automatically on every clock action (still works if location is denied).
- Optionally take a selfie at clock time — or skip it; it never blocks attendance.
- See a live clock, a location chip, a status dot, today's total hours and a timeline of today's clock events.
- Add hourly time-log blocks (From / To / what they did) and delete their own entries.
- Write, edit and resubmit today's work report (work done / progress / blockers).
- See today's session summaries from the clock-out wrap-ups.

**Leave**
- See current-year balance cards for casual, sick and earned — remaining, used and pending.
- Apply for leave (type, from/to dates, free-text reason), with the balance guard enforced server-side and the error shown verbatim.
- Review their last 60 requests with colour-coded status pills.
- Cancel any still-pending request.

**Money**
- See wallet balance, pending withdrawal and available balance.
- Add or change a bank account (holder, account number, IFSC, bank name, optional UPI id) — the newest becomes primary; numbers display masked.
- Request a withdrawal up to their available balance, with an optional note.
- Track withdrawal status (pending / approved / paid / rejected / failed).
- Read the last 20 wallet ledger entries.
- See the last 12 payslips with month/year and net amount — **but the Download button does not work for them** (it points into an admin-only route).

**What they cannot do for themselves**
- Request an attendance correction or regularisation.
- See who approved their leave, or the approver's note.
- See a holiday calendar or a team leave calendar.
- View their own salary structure, submit tax declarations or investment proofs, or actually open a payslip.
- Cancel a pending withdrawal.
- See, confirm or contest their own worker classification.
- Raise or track a hiring requisition.
- Receive any notification about any of the above — every module is pull-only.

## 5. What is NOT built / half-built

### Cross-cutting (true of the whole HRMS)
- **No API layer for any HR module.** Everything is an inline Astro form POST. No mobile app, kiosk, biometric device or external integration can read or write attendance, leave, payroll, wallet, classification or requisitions.
- **No notifications, anywhere.** Not one HR module calls `notify()` or `push`. The `payroll_run` notification type is fully registered across `notification-catalog.ts`, `notify-audience.ts`, `push.ts` and `notify.ts` — and never emitted.
- **No audit trail.** Nothing in `src/pages/admin/hr` writes to `audit_log` via `logAudit`. All provenance is denormalised, overwrite-in-place columns (`decided_by`, `approved_by`, `classification_reviewed_by`).
- **Most HR tables have no DDL in the repo.** `hr_attendance`, `hr_payroll_runs`, `hr_payslips`, `hr_salary_structures`, `hr_leave_types`, `hr_leave_balances`, `hr_leave_requests` exist only in production. `src/lib/db/schema.ts` has no `hr_*` tables at all.
- **Schema DDL runs on user request paths.** `ensureLeaveSchema`, `ensureWalletSchema`, `ensureRequisitionSchema`, `ensureClassificationSchema`, the `CREATE TABLE hr_task_log` inside the clock-out handler, and the `ALTER TABLE … ADD COLUMN face_photo` in the same handler.
- **Silent failure is the default.** `safe()` helpers and empty catches mean a DB outage renders as an empty, healthy-looking screen — "No leave requests waiting.", a ₹0.00 wallet, an unmarked attendance day.
- **No CSRF tokens on any POST form** on either surface.
- **The `hr` role gate is a substring test** (`role.indexOf('hr') >= 0`) in `approverRole`, the leave page and the wallet page — any future role whose name merely contains "hr" silently gains org-wide leave approval and payout-release rights.
- **`approverRole` uses string-interpolated SQL** (`sql.raw`, quotes stripped by hand at `src/lib/hr-wallet.ts:115`) instead of bound parameters, for both leave and wallet decisions.
- **No holiday calendar exists anywhere in the system.** "holiday" is only a selectable attendance status.
- **The modules are decoupled.** Approved leave does not create an `on_leave` attendance row. Unpaid leave does not reduce pay. Attendance does not prorate payroll. Payroll does not credit the wallet. Classification changes nothing downstream.

### Leave Management
- **Two incompatible leave schemas are live at once.** The working flow writes `hr_leave_request` (singular); the employee profile Leave tab and *both* dashboard "pending leave" KPIs read `hr_leave_requests` / `hr_leave_types` / `hr_leave_balances`. A request filed in the portal will never appear in those counters or on the employee's own profile.
- `hr_leave_types` is never seeded by any code, so on a fresh DB `syncHiredToEmployee` allocates zero balance rows and the Leave tab is permanently empty.
- **No accrual engine.** The word "accrual" does not exist in the module. Full-year allowance granted on 1 January; no pro-rata, no mid-year-joiner proration, no probation hold, no carry-forward job, no encashment, no year-end rollover. `carried_forward` is read but never written.
- **Weekends and holidays are debited.** `daysBetween` is a raw inclusive calendar-day count.
- **No half-day or hourly leave** — `days` is `INT`, even though attendance has a `half_day` status.
- **No overlap or duplicate detection.** An employee can stack overlapping requests until the aggregate balance check trips.
- **The reporting-manager approval path has no UI.** `approverRole` can return `reporting_manager`, but the only page that calls `decideLeave` redirects non-admin/non-hr users away. Conversely the HR queue is unscoped — every HR user sees and can decide every request.
- **Year-boundary bug.** A 28 Dec – 4 Jan request debits all 8 days from the starting year.
- No history table — decisions are overwritten in place. No status CHECK, no FK to `hr_employees`.

### Attendance & Time Tracking
- **No face matching.** The selfie is stored and eyeballed ("advisory — verify visually"); it is never compared to the face-enrolment record used for 2FA. No score, no flag, no reject path.
- **No geofencing.** lat/lon/accuracy are captured on every event and compared to nothing — no office-location table, no radius. Worse, **the coordinates are never displayed on any admin screen**; `location_name` is selected but not rendered.
- **Breaks are never deducted.** `work_hours` is always clock-out minus clock-in. Break time is fully paid.
- **`overtime_hours` is read but never written** by any code, and the monthly tile that sums it isn't even rendered. No shift definitions, no expected-hours baseline — overtime and lateness cannot be derived.
- **No approval or regularisation workflow.** Any admin with the `attendance` permission can silently rewrite or hard-DELETE any employee's day: no reason field, no approval, no audit row, no soft delete. Employees cannot request a correction.
- **Work mode is silently downgraded by an edit** — `field` is offered at clock-in but absent from both admin selects, so editing that day overwrites it.
- **Timezone inconsistency.** The portal computes "today" as the UTC date; the admin renders in Asia/Kolkata. Between 00:00 and 05:30 IST, events attach to the previous calendar date.
- **The overnight-shift heal UPDATE runs on every page view**, not as a migration, with errors swallowed.
- `hr_time_logs` has no overlap or total-hours validation, cannot represent an overnight block (`varchar(5)` HH:MM with end > start), is never reconciled against clocked hours, and admins can view but not edit or delete entries.
- `hr_daily_reports.hours` exists in the migration and is never written or read.
- **No reminders or enforcement** — the "N not yet marked" banner and "No daily report submitted" line are display-only.
- **No reporting or export.** No CSV/XLSX, no calendar grid, no team or department view. The Monthly tab is one employee at a time and hardcoded to `DATE_TRUNC('month', NOW())` — past months cannot be viewed at all.
- Portal `hr_attendance` upsert and update are both `.catch(() => {})`, so a clock event can land in `hr_clock_events` while the attendance row silently fails — and the employee still sees "Clocked in successfully!".
- Latent bug at `src/pages/admin/hr/employees/[id].astro:338`: `const attendanceRows = (Array.isArray(attendance) ? attendance : attendanceRows) as any[];` — the fallback branch references the variable being declared and would throw a ReferenceError; it only survives because postgres-js always returns an array.

### Payroll & Payslips
- **The employee payslip download is broken.** `/portal/employee` links to `/admin/hr/payslip/{id}`, which middleware redirects away from and the route itself 401s unless the role is `super_admin`/`hr`. There is no portal-scoped payslip route.
- **No actual payment.** `mark_paid` only flips two status columns. No bank transfer, no NEFT/bank-advice file, no UTR field — and no link to the RazorpayX payout code that already exists in `src/lib/hr-wallet.ts`.
- **No prorating or loss of pay.** `days_worked` / `days_leave` / `days_absent` are stored and shown but never enter the calculation. An employee absent the entire month is paid in full.
- **Statutory rules are hardcoded page constants** (PF 12% capped 1800, ESIC 0.75% below 21000, PT 200/150 slabs). No config table, no financial-year versioning, not editable by an admin.
- **Currency is cosmetic.** The same rupee thresholds are applied to USD/GBP/EUR/SGD/AED runs, and the success message hardcodes `₹`.
- **No employer-side contributions stored** — no employer PF, employer ESIC or gratuity column anywhere, so CTC and employer liability cannot be reported.
- **No TDS computation.** `tds` is a flat number typed by HR. No slabs, no old/new regime, no 80C or HRA exemption, no investment declarations or proofs, no Form 16 / 24Q.
- **No statutory filing outputs** — no PF ECR, no ESIC challan, no PT return, no UAN/ESIC numbers, and no PAN on the payslip even though `hr_employees` stores `pan_number`.
- `other_deductions` is read and printed but never written by the salary form. **There is no per-payslip adjustment UI at all** — no bonus, arrears, overtime, reimbursement or one-off deduction after a run is generated.
- **A run cannot be deleted, re-run or reverted.** No `delete_run`, no `recalculate`; duplicate month+year is rejected, so a run built on wrong salary data is permanently stuck. Approve and mark-paid are both one-way.
- **No transaction boundary and an N+1 loop.** A mid-loop failure leaves a partial run with wrong totals, caught only by a generic `catch`.
- **An employee with no salary structure is silently paid on invented numbers** (HRA 40%, transport 1600, medical 1250) with no flag and no dry-run preview.
- No PDF generation, no bulk payslip export, no CSV/Excel salary register, no email delivery.
- **Coarse authorization.** `userCanAccess(userId, pageKey, 'edit'|'export')` exists in `src/lib/auth/permissions.ts` and is never called by the payroll page — view access implies full write. **No maker-checker**: the same user can create, approve and mark a run paid.
- **Payslip HTML interpolates DB values unescaped** (`full_name`, `designation`, `employee_code`, `dept_name`, `currency`) — stored HTML renders as markup.
- Salary setup is capped at `LIMIT 100` with no pagination or search — payroll cannot be configured for more than 100 employees.
- **No multi-entity / multi-country payroll**: no legal-entity dimension, no pay cycle other than calendar month, no off-cycle or final-settlement run, and run uniqueness is global on month+year.
- The `payouts` section key is mapped in `middleware.ts:23` and listed in `admin-sections.ts`, but `src/pages/admin/hr/payouts` does not exist — a dead route.

### HR Wallet / Payouts
- **No payout reconciliation.** `payWithdrawal` marks the row `paid` as soon as Razorpay returns an id, but RazorpayX payouts are asynchronous (queued/processing/reversed). No webhook, no polling job, and **nothing in the codebase ever writes the `failed` status** that both the schema comment and the admin filter anticipate.
- **No idempotency key on the payout call.** A timeout followed by a retry of `pay` can issue a duplicate bank transfer.
- `hr_bank_account.rzp_fund_account_id` is in the schema and never written or read — a fresh `fund_account` is inlined on every payout.
- `hr_bank_account.verified` defaults false, is selected, and is never set true. **No penny-drop or name-match verification, and `payWithdrawal` does not check it before paying.**
- **UPI payouts are not implemented.** `method` is always `'bank'`; `payWithdrawal` hardcodes `mode: 'IMPS'` / `account_type: 'bank_account'` and requires an account number — so a UPI-only account silently falls through to the manual path even though both UIs show `upi_id` as a payout target.
- **Payroll is not wired to the wallet.** `/admin/hr/payroll` never imports `hr-wallet` or calls `credit()`. Salary must be retyped by hand into "Credit a wallet", with no link back to the payslip.
- **No `debit()` export** — recoveries, advances and reversals are impossible without raw SQL.
- **No reversal or cancellation.** An employee cannot cancel their own pending request; a rejection cannot be reopened; a wrong credit cannot be undone through any UI.
- **Race condition on available balance.** `requestWithdrawal` reads then inserts with no transaction, row lock or DB constraint — two concurrent requests can jointly overdraw.
- **`/admin/hr/wallet` is not linked from any navigation.** It renders with `activeNav=""` and no `href` to it exists anywhere in `src/`. URL-only.
- **No maker-checker on credits** — a single HR user can mint wallet balance unilaterally, with no server-side upper bound on the amount.
- **No tests.** There is no `hr-wallet.test.ts` (unlike `src/lib/hr-onboarding.test.ts`), so the balance arithmetic, the available-balance guard and the approver chain are unverified.
- `getBalance` hardcodes `currency: 'INR'` and sums across all rows regardless of each row's `currency`.
- No pagination or server-side filtering — `LIMIT 100` and the function's `status` option is ignored in favour of client-side JS.
- The admin listing selects the raw unmasked `b.account_number` into page context even though only a masked form is ever rendered.

### Worker Classification
- **No criteria and no scoring.** Despite framing misclassification as the biggest compliance risk, there is no control test, no ABC test, no economic-reality factors and no per-person risk computation. `risk` is a static string on the *category*, so every contractor reads "high" and every permanent employee reads "low" regardless of the facts. **Nothing detects a contractor who is in practice an employee.**
- **No evidence capture.** No place to attach the consulting agreement, contractor invoices, the EOR contract, or a written rationale. No `classification_notes` column, no document link. If a labour authority asks "why is this person a contractor", the system has nothing.
- **No audit history.** `classification_reviewed_at`/`_by` are overwritten in place; a `permanent → contractor` change leaves no trace of the prior value.
- **No review enforcement.** The page says "Review at least once a year" and prints `never`, but there is no due-date computation, no sort/filter by stale review, no reminder job, no cron entry. Honour system.
- **Weak authorization.** The only check is `user.role === 'applicant'`. Any other authenticated admin-side role can change any worker's legal and benefits status.
- **No server-side validation.** `classification`, `country` and `engaged_via` are written straight from FormData with no check against the constant lists — a crafted POST writes an arbitrary string, and the UI then silently falls back to rendering it as `permanent`.
- **Two overlapping, unreconciled concepts.** `hr_employees.employment_type` (free text, values `Full-Time | Part-Time | Contract | …`, set by `transferApplicantToEmployee`) is edited on a different screen and never synced with `classification`. Nothing flags `employment_type='Full-Time'` + `classification='contractor'`.
- **No downstream consequence.** `classification` is read by nothing else in the codebase. Marking someone `eor` or `contractor` changes nothing in payroll, leave, wallet, onboarding or separation. It is a register, not a control.
- `LIMIT 300`, no pagination, no search, no view of inactive employees — so **the roll-up counts are counts of the first 300 rows, not of the workforce**.
- No CSV/export for counsel or an auditor, no API route, and no bulk action (moving 20 people to an EOR partner is 20 separate form submissions).
- `classification_reviewed_by UUID` has no `REFERENCES users(id)` — reviewer identity is unenforced.

### Hiring Requisitions
- **The link to roles/openings is not implemented — this is the central gap.** `opened_role_id` is declared in the CREATE TABLE, commented "references roles(id) once role is created from this req", and is never written, read or selected anywhere. There is no "Create role from requisition" action, nothing flips `roles.is_open`, and there is no FK. `roles.isOpen` defaults to `true` and `/admin/roles` can open a role with no requisition at all. **An approved requisition is a document, not a trigger** — the page copy "Only approved requisitions become open roles" describes intent the code does not enforce.
- **No authorization on who approves.** `decideRequisition()` takes a `userId` and never checks a role; the page gate is "signed in and not an applicant". Any user who can load the page can approve at Finance and then approve again at Leadership — **including the person who raised the requisition**. No self-approval block, no separation of duties.
- **No status guard on the UPDATE.** Every statement is `WHERE id=…` with no `AND status=…`, and `level` arrives as a client-controlled hidden field. A replayed or hand-crafted POST can stamp finance approval onto an already-approved requisition, or reject an approved one. With no CSRF tokens anywhere, the only protection is the session cookie's `sameSite='lax'`.
- **No third (CPO) approval stage**, despite the module comment promising "Finance + Leadership + (CPO once we have one)". The chain is a hardcoded two-branch `if`.
- The `withdrawn` status is in the schema's status list but there is no withdraw action in the UI or the library.
- **No employee/manager surface at all.** A hiring manager without admin `hr` access cannot raise or track a requisition, and receives no notification of the outcome.
- No detail route, no edit route, no API. Register is `SELECT *` capped at `LIMIT 200` with no pagination.
- `CLASSIFICATIONS` is imported into the requisitions page and never used — the form uses its own separate `ENGAGEMENT_TYPES` constant, so the intended link between "what we are hiring for" and "how the person will be classified" is not actually wired.
- `budget_currency` defaults to `'CHF'` while the rest of the money stack is INR-first.

### Whole capability areas with nothing built
- **People development** — no performance reviews, goals/OKRs, appraisals, promotions, competencies, training, LMS linkage or career paths. `kras` is captured once on a requisition and consumed by nothing.
- **Exit / offboarding** — no resignation flow, notice-period tracking, clearance workflow, exit interview, final settlement run, leave encashment, F&F statement, relieving letter or asset return. The only artefact is a manual `final_payroll` boolean on a checklist in `src/pages/admin/hr/employees/[id].astro`.
- **Org & policy foundations** — no holiday calendar, no shift/roster definitions, no leave policy engine, no statutory-rate configuration table, no legal-entity/company dimension.
